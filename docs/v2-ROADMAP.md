# Story-to-Game 平台化路线图

**日期**：2026-07-07
**版本**：v5.1（平台化 MVP 阶段）
**定位**：在线 AI 互动叙事平台
**核心原则**：静态前端 + 轻量 API + JSON 文件存储，无数据库、无用户系统

---

## 1. 产品定位

> **「在线 AI 互动叙事平台」—— 粘贴小说，AI 自动生成分支剧情游戏，即时游玩**

### 1.1 核心转变

| 维度 | 旧定位（工具套件） | 新定位（在线平台） |
|------|---------------------|---------------------|
| 产品形态 | 开源 HTML/JS 工具套件 | 在线平台 + API 服务 |
| 使用方式 | 下载 HTML → 手写 JSON → 本地运行 | 粘贴小说 → AI 生成 → 即时游玩 |
| 存储方案 | 无后端，本地文件 | 服务端 JSON 文件存储（Vercel Blob / 本地文件） |
| AI 能力 | SKILL.md 作为文档参考 | API 服务自动调用 AI 生成 |
| 创作门槛 | 需理解 JSON 格式，手动编辑 | 上传小说即可，AI 全自动生成 |
| 部署方式 | GitHub Release 分发 | Vercel 全栈部署 + 国内云双轨 |
| 播放器方案 | 原生 HTML/JS 启动器 | iframe 嵌入原生引擎（保留 100% 功能） |

### 1.2 架构概览

```
┌─────────────────────────────────────────────────────────────┐
│                    前端层（Next.js App Router）                │
│  ├─ 首页（产品介绍 + CTA）                                    │
│  ├─ 即时体验页（粘贴 → 生成 → 游玩，含规则配置）              │
│  ├─ 作品库页（列表 + 分类 + 点击即玩）                        │
│  └─ 播放器页（iframe 嵌入原生 JS 引擎）                       │
├─────────────────────────────────────────────────────────────┤
│                    API 层（Next.js Route Handlers）            │
│  ├─ POST /api/generate       — SSE 流式 AI 生成              │
│  ├─ POST /api/works          — 保存作品                      │
│  ├─ GET  /api/works          — 作品列表                       │
│  └─ GET  /api/works/:id      — 获取单个作品 JSON              │
├─────────────────────────────────────────────────────────────┤
│                    AI 层（多提供商）                            │
│  ├─ 硅基流动 / DeepSeek / 百炼 / OpenAI / OpenRouter         │
│  └─ 环境变量切换，OpenAI 兼容格式                             │
├─────────────────────────────────────────────────────────────┤
│                    存储层（抽象适配）                            │
│  ├─ Vercel Blob（国际部署）                                   │
│  ├─ 本地文件系统（开发环境）                                   │
│  └─ 阿里云 OSS / 腾讯云 COS（国内部署预留）                    │
└─────────────────────────────────────────────────────────────┘
```

**技术栈**：Next.js 14 App Router + TypeScript + Tailwind CSS + Shadcn/ui

### 1.3 不变的核心原则

以下原则从项目诞生起持续有效：

1. **文学优先**：每一个选择必须从当前场景自然长出，每一个后果必须被世界认真承接，每一个结局必须以判词收束
2. **轻 RPG 服务叙事**：数值变化先有文本回响，再展示数值，不压过文学体验
3. **JSON 可校验**：所有 JSON 必须通过 validate.py 的 18 条 RPG 校验规则
4. **向后兼容**：新增字段均为可选，旧 JSON 不受影响（`meta.rpg.enabled=false` 维持 v1.x 行为）
5. **类型深度**：修仙/无限流/悬疑/末世/宫斗各有专属模板、状态系统、主题色和 VFX
6. **选择有重量**：choice.weight 四级系统 + delayedChanges 三层后果
7. **成长有仪式**：milestones 三级庆祝 + endings 多类型结局

### 3.4 风险检查点

| 风险 | 概率 | 影响 | 缓解策略 | 触发条件 |
|------|------|------|----------|----------|
| AI 生成质量不稳定 | 高 | 高 | 快速模式先上，人工审核兜底；校验器拦截不合规 JSON；rules 参数引导 | validate 通过率 < 70% |

| 修仙 Demo 生成效果差 | 中 | 高 | 准备手工精编的 fallback Demo；迭代 prompt；自定义 rules 引导 | 用户测试满意度 < 6/10 |

## 7. 风险总览

### 7.1 全局风险

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| AI 生成质量无法满足用户期望 | 高 | 致命 | 快速模式先行+人工兜底；持续迭代 prompt；rules 参数精细引导；建立示例库 |
| AI API 成本随规模增长失控 | 中 | 高 | Token 配额管理；结果缓存；分模式定价回收成本；多提供商比价 |
| 竞品跟进（橙光/易次元上线 AI 功能） | 中 | 高 | 先发优势+类型深度壁垒；SKILL.md 九步工作流质量壁垒 |
| 无用户系统的功能限制 | 中 | 中 | 按需评估引入轻量登录（微信授权）；社区功能匿名优先 |
| JSON 文件存储不支撑规模 | 低 | 中 | Q4 末评估是否需要引入数据库；JSON 可平滑迁移 |
| 内容合规风险 | 中 | 高 | AI 审核+人工复审+举报机制；敏感内容类型优先审核 |

### 7.2 Plan B：核心假设不成立时的备选方案

| 假设失败 | 转向方案 |
|----------|----------|
| AI 生成质量不达标 | 转向"AI 辅助 + 人工精编"模式，AI 负责结构框架，创作者编辑细节 |
| 在线平台用户增长慢 | 保留 API 服务，同时恢复开源工具套件模式，双轨并行 |
| 修仙/网文类型市场太小 | 扩展到通用叙事（剧本杀、教育互动、品牌营销互动） |
| 微信小程序审核持续被拒 | 聚焦 Web 站 + PWA，放弃小程序渠道 |

## 8. 利益相关者视图

### 8.1 产品视图

- **定位**：在线 AI 互动叙事平台
- **核心价值**：零门槛创作（粘贴小说即可）+ 即时游玩 + AI 全自动生成
- **差异化**：类型深度（5 种专属模板）+ 文学保真（九步工作流）+ 轻 RPG（服务叙事不压文学）+ 自定义规则（10 项创作者控制）
- **时间表**：Q3 MVP → Q4 分发 → Q1 生态 → Q2 商业化

### 8.2 技术视图

- **技术栈**：Next.js 14 App Router + TypeScript + Tailwind CSS + Shadcn/ui
- **架构约束**：无数据库、无用户系统、静态前端 + 轻量 API
- **关键决策**：iframe 嵌入原生引擎（保留 100% 功能）；多提供商 AI 客户端；JSON 文件存储可平滑迁移
- **扩展路径**：JSON 文件存储 → 按需引入数据库；无用户系统 → 按需引入微信授权登录

### 8.3 设计视图

- **设计语言**：Glassmorphism 暗色系，5 种题材主题实时切换
- **设计原则**：文学优先，数值不压过内容
- **动效原则**：题材专属 VFX + 粒子特效（可关闭），里程碑全屏庆祝动画

---

> 本路线图将 Story-to-Game 从"开源工具套件"升级为"在线 AI 互动叙事平台"，技术栈全面迁移至 Next.js 生态。核心路径：Q3 验证 MVP → Q4 扩大分发 → Q1 丰富生态 → Q2 实现商业化。每季度末有可验证的交付标准和明确的假设检验节点。技术细节参见 v2-IMPLEMENTATION.md。
