# Story-to-Game v2.0 调整路线图

**日期**：2026-07-02
**版本**：v2.0
**基于**：原 ROADMAP.md + PRD v1 + 竞品分析 交叉验证
**调整说明**：将原 10 阶段重组为 4 季度迭代，明确验证节点，增加并行工作流

---

## 1. 原路线图问题与优化

### 1.1 原路线图（10 阶段）问题

| 问题 | 说明 | 优化方案 |
|------|------|----------|
| 阶段过细 | 10 个阶段，每阶段交付价值不清晰 | 合并为 4 季度，每季度有明确验证目标 |
| 顺序执行 | 未考虑并行，总周期 9-12 个月 | 开发/内容/文档可并行 |
| MVP 模糊 | 阶段 1-2 是 MVP，但未明确验证什么假设 | Q3 明确验证「修仙 + 轻 RPG」假设 |
| 缺少商业化 | 只到阶段 9，无商业化规划 | 2027 Q2 增加商业化探索 |
| 缺少移动端 | 只做 HTML，未规划移动端 | 2027 Q1 评估移动端方案 |
| 无反馈循环 | 无用户测试节点 | 每季度末安排用户验证 |

### 1.2 优化原则

```
1. 聚焦：Q3 只做一件事——验证「修仙 + 轻 RPG」
2. 并行：开发 RPG UI 时，内容顾问并行写 Demo
3. 验证：每季度末必须有用户反馈或数据验证
4. 兜底：如果假设不成立，有明确的 Plan B
5. 渐进：新功能必须向后兼容，旧 JSON 不受影响
```

## 3. Q3 2026：MVP 验证（7-9 月）

### 3.1 季度目标

验证核心假设：**「修仙小说的境界体系适合轻 RPG 化，且不会干扰文学体验」**

### 3.2 并行工作流

```
Week 1-2:
  [开发] P0-1: meta.rpg schema 定义 ──┐
  [开发] P0-2: 顶部状态栏 UI          │ 串行（schema → UI）
  [文档] 修仙模板规范撰写            ──┤ 可与开发并行
                                       │
Week 3-4:
  [开发] P0-3: 状态详情抽屉          ──┤
  [开发] P0-4: 变化反馈 Toast        ──┤ 串行
  [内容] 修仙 Demo 故事大纲          ──┤ 可与开发并行
                                       │
Week 5-6:
  [开发] P0-5: 条件选项置灰          ──┤
  [开发] P0-7: validate.py 增强      ──┤ 可并行
  [内容] 修仙 Demo JSON 撰写         ──┤ 需要等 schema 稳定
                                       │
Week 7-8:
  [联调] 修仙 Demo + 播放器联调      ──┤
  [测试] 内部测试 + bug 修复         ──┤
                                       │
Week 9-10:
  [用户] 5-10 名目标用户测试         ──┤
  [迭代] 根据反馈修复                ──┤
  [发布] GitHub Release v0.2         ──┤
                                       │
Week 11-12:
  [市场] B 站/小红书演示视频         ──┤
  [社区] README 更新 + 宣传          ──┤
```

### 3.3 里程碑

| 里程碑 | 时间 | 交付物 | 成功指标 |
|--------|------|--------|----------|
| M1.1 | 7 月底 | `meta.rpg` schema + 状态栏 UI | 修仙 Demo 可显示境界/修为 |
| M1.2 | 8 月中 | 变化反馈 + 条件选项 | 选择后显示「灵力 +10」 |
| M1.3 | 8 月底 | 修仙 Demo 可玩 | 40-80 节点，3-5 结局 |
| M1.4 | 9 月中 | 用户测试报告 | 满意度 ≥ 7/10 |
| M1.5 | 9 月底 | GitHub Release v0.2 | Stars > 100 |

### 3.4 成功指标

- 修仙 Demo 可完整游玩，文学保真度得到作者认可
- RPG 数值不干扰阅读体验（负评率 < 15%）
- GitHub Stars > 100
- 至少 1 个外部贡献者或 Issue

### 3.5 Plan B

如果「轻 RPG 干扰文学性」假设不成立：
- 转向纯文学模式（`meta.rpg.enabled=false` 为默认）
- RPG 作为可选功能，而非核心卖点
- 重点转向「AI 改编质量」和「单文件部署」

## 6. 2027 Q2：平台化探索（4-6 月）

### 6.1 季度目标

探索商业化路径，验证「创作者愿意为工具付费」假设

### 6.2 可选方向

**方向 A：自建创作者平台**
- 上传/管理作品
- 嵌入支付（微信支付/支付宝）
- 数据分析（游玩次数、完成率、选择分布）

**方向 B：与橙光/易次元合作**
- 作为「AI 改编工具」集成到现有平台
- 分成模式：作者付费使用 AI 改编功能

**方向 C：OpenCore + 捐赠**
- 基础功能免费开源
- 高级功能（批量生成、云存储、数据分析）付费
- GitHub Sponsors + 爱发电

### 6.3 成功指标

- 商业化路径确定并开始执行
- 有稳定收入覆盖基础运营成本

## 9. 利益相关者沟通

### 9.1 高管视图

- **定位**：类型小说 AI 互动改编工具，抢占 320 亿互动阅读市场
- **差异化**：文学保真 + 类型深度 + 单文件部署
- **资源**：1 名全栈 + 1 名内容顾问，或 2 名全栈
- **时间表**：12 个月达到商业化准备状态

### 9.2 工程视图

- **技术栈**：HTML/JS（单文件），无后端（除非做平台化）
- **关键决策**：轻 RPG（不做复杂公式），JSON 可校验
- **并行点**：开发 RPG UI 时，内容顾问可并行撰写 Demo
- **技术债务**：单文件 HTML 需拆分为源码模块（Q3 完成）

### 9.3 设计视图

- **UI 原则**：文学优先，数值不压过内容
- **设计语言**：暗色系，支持明/暗切换
- **动效原则**：轻量，用户可选关闭
- **移动端**：响应式优先，APP 备选
