**日期**：2026-07-04

**调整说明**：将原 10 阶段重组为 4 季度迭代，明确验证节点，增加并行工作流。整合UI V2设计规范、选择重量系统、渐进探索、成长里程碑、三层后果、结局数据包

## 1. 原路线图问题与优化

### 1.1 原路线图（10 阶段）问题

| 问题 | 说明 | 优化方案 |
|------|------|----------|
| 阶段过细 | 10 个阶段，每阶段交付价值不清晰 | 合并为 4 季度，每季度有明确验证目标 |
| 顺序执行 | 未考虑并行，总周期 9-12 个月 | 开发/内容/文档可并行 |
| MVP 模糊 | 阶段 1-2 是 MVP，但未明确验证什么假设 | Q3 明确验证「修仙 + 轻 RPG」假设 |
| 缺少商业化 | 只到阶段 9，无商业化规划 | 2027 Q2 增加商业化探索 |
| 缺少移动端 | 只做 HTML，未规划微信生态入口 | Q3 技术验证小程序，Q4 进入内测 |
| 无反馈循环 | 无用户测试节点 | 每季度末安排用户验证 |

### 1.2 优化原则

```
1. 聚焦：Q3 产品主线只验证「修仙 + 轻 RPG」
2. 并行：开发 RPG UI 时，内容并行写 Demo
3. 分端：HTML 负责开放部署，小程序负责移动传播
4. 验证：每季度末必须有用户反馈或数据验证
5. 兜底：如果假设不成立，有明确的 Plan B
6. 渐进：新功能必须向后兼容，旧 JSON 不受影响
```

### 3.4 成功指标

- 至少 1 个外部贡献者或 Issue

### 3.5 MVP 范围

最小可行版本只做以下 5 件事：

1. `meta.rpg.primaryStats` — 定义可见数值结构。
2. 状态栏展示 — 让玩家看到境界、修为等关键状态。
3. `changes` 前后 diff 弹窗 — 选择后显示数值变化。
4. 条件选项置灰提示 — 不满足条件时展示原因。
5. 修仙 Demo — 40-50 节点，验证核心玩法假设。

小程序在 MVP 阶段只作为技术验证并行推进：抽出共享 core，验证同一份修仙 Demo JSON 能在小程序端播放，不进入产品主线范围。

这 5 件事不大改结构，但能让体验从「分支小说」明显变成「轻 RPG 文游」。

### 3.6 Plan B

如果「轻 RPG 干扰文学性」假设不成立：
- 转向纯文学模式（`meta.rpg.enabled=false` 为默认）
- RPG 作为可选功能，而非核心卖点
- 重点转向「AI 改编质量」和「单文件部署」

## 7. 风险矩阵

| 风险 | 概率 | 影响 | 缓解策略 | 触发条件 |
|------|------|------|----------|----------|
| 用户验证失败 | 中 | 高 | Q3 尽早测试；Plan B 转向纯文学 | 满意度 < 6/10 |
| 技术实现困难 | 低 | 高 | 坚持轻 RPG，数值 <= 5 | 开发周期超 2 倍 |
| AI 生成不稳定 | 中 | 中 | 校验 + 人工编辑兜底 | validate 通过率 < 70% |
| 竞品跟进 | 中 | 高 | 先发优势 + 类型深度 | 橙光/易次元宣布 AI 功能 |
| 小程序审核风险 | 中 | 高 | 早期 Demo 内容克制，补齐协议和声明 | 小程序提交审核被拒 |
| 小程序维护成本上升 | 中 | 中 | 共享 core，UI 分端实现 | 双端同类 bug 重复出现 |
| 资源不足 | 高 | 高 | 开源社区 + 寻找合作者 | 开发停滞 > 2 周 |
| UI V2 组件与 JSON 驱动对接工作量超预期 | 中 | 高 | P2 优先聚焦 JSON 驱动对接，非功能美化 | 开发进度滞后 2 周+ |
| delayedChanges 调试复杂 | 中 | 中 | 在调试器中增加 delayedChanges 追踪面板 | 用户报告延迟后果不符合预期 |
| VFX 在低端设备卡顿 | 中 | 中 | 检测 GPU 性能，自动降级到无粒子模式 | 动画帧率低于 30fps |

### 7.1 风险控制细则

**数值压过文学**

- 可见数值最多 5 个。
- 每个数值必须有叙事含义。
- 不做复杂公式。
- 选择反馈先写文本，再显示数值。

**JSON 变复杂**

- 所有新字段可选。
- 旧 JSON 完全兼容。
- `interactions` / `inventory` 分阶段加入。
- `validate.py` 给出清晰错误。

**AI 生成失控**

- Skill 提供类型模板。
- 每类限制核心数值数量。
- 每类规定最小互动结构。
- 校验器检查结构。
- 示例库提供参考。

**启动器难维护**

- 源码拆分开发。
- 构建时打包单文件。
- 发布物仍保持「一个 HTML 可运行」。
- 共享剧情 core 后，HTML 和小程序复用同一套逻辑。

## 9. 利益相关者沟通

### 9.1 高管视图

- **定位**：类型小说 AI 互动改编工具，抢占 320 亿互动阅读市场
- **差异化**：文学保真 + 类型深度 + 单文件部署
- **时间表**：12 个月达到商业化准备状态

### 9.2 工程视图

- **技术栈**：HTML/JS（单文件）+ 微信小程序原生渲染，短期无后端，Q4 起可接入云端作品库
- **关键决策**：轻 RPG（不做复杂公式），JSON 可校验
- **并行点**：开发 RPG UI 时，内容可并行撰写 Demo
- **技术债务**：单文件 HTML 需拆分为源码模块，并抽出共享剧情 core

### 9.3 设计视图

- **UI 原则**：文学优先，数值不压过内容
- **设计语言**：Glassmorphism（毛玻璃）沉浸式暗色系，5 种题材主题实时切换
- **动效原则**：题材专属 VFX + 粒子特效（可关闭），章节转场 1.5 秒统一时长
- **移动端**：微信小程序优先，响应式 HTML 保留，APP 远期备选
- **UI V2状态**：UI V2已完成原型设计（含10个组件、5种题材主题、VFX特效），P2核心任务是把已有UI组件从硬编码数据切换为JSON驱动
