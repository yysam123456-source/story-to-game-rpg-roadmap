# 微信小程序并行集成方案

本文档补充 Story-to-Game v2.0 的移动端策略：HTML 播放器继续作为开源、离线和独立部署主阵地，微信小程序作为中文用户传播、移动端体验和未来商业化的并行入口。

## 集成结论

微信小程序可以并行推进，但不建议把现有单文件 HTML 直接作为长期主方案嵌入小程序。推荐路径是抽出平台无关的剧情运行核心，再分别由 HTML 播放器和微信小程序播放器渲染同一份 JSON。

```text
AI Skill / JSON 剧本协议
        ↓
共享剧情运行核心
        ↓
├── HTML 播放器
└── 微信小程序播放器
```

## 集成方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| `web-view` 嵌入 HTML | 将 HTML 部署到 HTTPS 域名，小程序通过 `web-view` 打开 | 上线最快，改造少 | 需要业务域名，离线优势消失，文件导入、存档、分享受限 | 只用于早期演示 |
| 原生小程序播放器 | 用 WXML/WXSS/JS 实现播放器 UI，消费同一份 JSON | 体验最好，可接入微信分享、存档、登录、支付 | 需要开发一套小程序渲染层 | 推荐主路线 |
| 跨端框架 | 用 Taro/uni-app 重写播放器并输出 Web/小程序 | 长期可减少端差异 | 改造成本高，当前单 HTML 迁移重 | 中长期评估 |

推荐采用“原生小程序播放器 + 共享核心逻辑”。`web-view` 可以作为短期演示入口，但不应成为正式产品架构。

## 产品分工

| 端 | 主要用途 | 关键能力 |
| --- | --- | --- |
| HTML 播放器 | 开源分发、教育场景、GitHub Pages、离线演示、独立部署 | 单文件运行、导入 JSON、本地存档 |
| 微信小程序 | 手机用户游玩、微信分享、未来登录/云存档/支付 | 原生 UI、分享卡片、云端作品库、微信生态 |
| 原生 APP | 远期备选 | 需要更复杂的资源、账号和商业化能力 |

## 架构调整

### 共享核心

现有启动器需要逐步拆出平台无关逻辑，避免 HTML 和小程序各自维护一套剧情推进规则。

```text
core/
├── story-engine.js       # 节点推进、next、choices、routes
├── condition-engine.js   # condition 判断，不使用 eval
├── change-engine.js      # changes 应用和差异计算
├── save-model.js         # 存档数据结构
├── validation-lite.js    # 运行时轻校验
└── adapters/
    ├── web-adapter.js
    └── wechat-adapter.js
```

### 双端渲染

```text
HTML 播放器
  -> 调用 core
  -> 使用 DOM/CSS 渲染

微信小程序播放器
  -> 调用 core
  -> 使用 WXML/WXSS 渲染
```

共享层只处理剧情状态，不处理页面布局。UI 各端独立实现，避免为追求跨端复用而牺牲交互体验。

## 小程序 MVP 范围

小程序早期版本只验证“同一份 JSON 能在微信环境中稳定播放”。它不抢占 Q3 HTML 修仙 MVP 的主线资源。

### Q3 技术验证

目标：用小程序播放同一份修仙 Demo JSON。

范围：

- 加载内置 Demo JSON。
- 渲染正文 `segments`。
- 渲染选项 `choices`。
- 展示 `meta.rpg.primaryStats` 状态栏。
- 执行 `changes` 并显示变化反馈。
- 支持条件选项置灰。
- 使用 `wx.setStorageSync` / `wx.getStorageSync` 保存本地进度。

不做：

- 不做登录。
- 不做支付。
- 不做云存储。
- 不做作品市场。
- 不做在线编辑器。
- 不做用户导入 JSON。

### Q4 内测版

目标：让微信小程序达到可邀请用户试玩的程度。

范围：

- 作品列表。
- 云端 JSON 加载。
- 微信分享卡片。
- `interactions` 场景交互。
- `inventory` 背包。
- 修仙 Demo + 无限恐怖 Demo。
- 基础数据记录：开始游玩、到达结局、退出节点。

### 2027 Q1 公开测试

目标：验证小程序作为分发端是否成立。

范围：

- 微信登录。
- 云存档。
- 收藏作品。
- 游玩统计。
- 创作者作品上传方案评估。
- 打赏、付费章节或订阅方案评估。

## 小程序页面规划

MVP 建议 5 个页面：

```text
pages/index
  作品列表 / 内置 Demo 入口

pages/player
  剧情播放主页面

pages/status
  RPG 状态详情

pages/archive
  存档 / 读档

pages/about
  作品信息 / 作者信息 / 版权说明
```

Q3 播放页不包含 `interactions` 区域；该区域在 Q4 加入，与 HTML 路线图保持一致。

## 数据与存档

HTML 端和小程序端应使用同一份存档模型。

```json
{
  "storyId": "qinglu-yehuo",
  "currentNodeId": "node_023",
  "val": 45,
  "variables": {},
  "flags": [],
  "inventory": {},
  "achievements": [],
  "visitedNodes": [],
  "playedAt": 1783000000
}
```

HTML 端使用 `localStorage`，小程序端使用微信本地存储。后续做云存档时，仍沿用该结构。

## JSON 加载策略

### 内置 Demo

Q3 技术验证使用内置 Demo，减少网络和审核变量。

```text
miniprogram/assets/stories/qinglu-yehuo.json
```

### 云端作品库

Q4 内测版引入云端作品库。

```text
小程序启动
  -> 拉取作品列表
  -> 下载 story.json
  -> 校验
  -> 缓存
  -> 播放
```

可选技术：

- 微信云开发。
- 静态 CDN。
- 自建 API。
- GitHub Pages 仅适合早期演示，不适合正式分发。

### 用户导入 JSON

小程序端用户导入 JSON 不作为早期目标。微信文件权限和体验不如 Web 端自然，适合在创作者工具成熟后再做。

## 条件表达式安全

小程序端不应使用 `eval`。所有条件都应走安全条件解析器。

兼容字符串条件：

```json
{
  "condition": "qi >= 20"
}
```

推荐对象条件：

```json
{
  "condition": {
    "all": [
      { "var": "qi", "op": ">=", "value": 20 },
      { "flag": "learned_basic_spell" }
    ]
  }
}
```

后续 Skill 应优先生成对象条件，字符串条件保留为兼容格式。

## 合规与审核

微信小程序比 HTML 分发更依赖内容审核。早期 Demo 应保持题材克制，避免因内容而影响产品验证。

需要补充：

- AI 生成内容声明。
- 用户协议。
- 隐私政策。
- 内容免责声明。
- 未成年人适龄提示。
- 作品版权说明。

内容约束：

- 避免过度血腥和恐怖描写。
- 避免现实政治、宗教敏感内容。
- 避免诱导付费或赌博式表达。
- 恐怖、宫斗、末世类型在小程序端需要更严格的文本审查。

## 路线图调整

小程序作为并行轨加入，不替代 HTML 主线。

```text
Q3 2026:
  HTML 主线：修仙 MVP
  小程序并行轨：共享 core 抽离 + 小程序技术验证

Q4 2026:
  HTML 主线：interactions + inventory + 无限恐怖 Demo
  小程序并行轨：云端作品列表 + 分享 + 内测版

2027 Q1:
  HTML 主线：多类型模板
  小程序并行轨：登录、云存档、公开测试

2027 Q2:
  平台化探索：小程序可作为优先商业化入口之一
```

## 验收标准

### Q3 技术验证

- 同一份修仙 Demo JSON 可在 HTML 和小程序中播放。
- 小程序端支持状态栏、选项、变化反馈和本地存档。
- 小程序端不使用 `eval`。
- 小程序端单次选择响应时间 ≤ 200ms。
- 小程序内置 Demo 包体可控，不影响提交审核。

### Q4 内测

- 小程序可展示作品列表。
- 小程序可从云端加载 JSON。
- 小程序可分享作品卡片。
- 修仙和无限恐怖两个 Demo 可完整游玩。
- 至少 10 名测试用户完成一次完整游玩。

## 风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 双端维护成本上升 | 开发速度下降 | 共享 core，UI 分端实现 |
| 小程序审核不通过 | 无法公开测试 | 早期 Demo 内容克制，补齐协议和声明 |
| `web-view` 体验差 | 用户留存差 | 只做演示，不作为长期方案 |
| 包体限制 | 无法内置大量作品 | Q3 内置 1 个 Demo，Q4 云端加载 |
| 条件表达式安全 | 审核和运行风险 | 使用安全解析器，不使用 `eval` |

## 推荐执行顺序

1. 抽出 `story-engine`、`condition-engine`、`change-engine`。
2. HTML 播放器改为调用共享 core。
3. 建立最小小程序项目，加载内置修仙 Demo。
4. 在小程序端实现播放页、状态栏、变化反馈和存档。
5. Q4 接入云端作品列表和分享。

微信小程序不应压缩 Q3 HTML MVP 的开发空间。Q3 的唯一产品验证目标仍是修仙轻 RPG 是否成立；小程序的 Q3 目标只是验证技术路径。
