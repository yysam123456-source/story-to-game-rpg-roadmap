# 叙事文本 → 分支剧情游戏 改写技能

## 概述

本技能将任意体量的线性叙事文本改写为符合「分支剧情游戏启动器」格式的 JSON 剧本。

本技能适用于 Claude、GPT、DeepSeek 及任何大语言模型。

## 最高创作原则

**每一个选择都必须从当前场景自然长出，每一个后果都必须被世界认真承接，每一个结局都必须在充分变化后以判词收束。**

这句话是所有规则的总纲。当任何具体规则之间产生冲突时，回到这句话判断。

具体展开：
- **自然长出**：选项不是"生成器填槽"，是此刻这个角色在这个处境下真正可能做的事
- **世界承接**：选择之后不是只改变量，是世界——人物态度、制度反应、物件归处、空间氛围——都被这个选择改变了
- **判词收束**：结局页不交代剧情，判词越短前面越要充分。ending 页像压下的印章，不是故事摘要
- **后果密度决定长短**：不是为了长而长，也不是为了短而短。一个选择改变了多少东西，就需要多少篇幅来展示这些改变。写到这些改变都被充分呈现了，自然停下

## 步骤强制执行（最高优先级铁律）

**所有规划步骤（记忆索引、风格提取、结构拆解、分支文学大纲、状态系统、章节设计文档）都是写给 AI 自己看的内部工作文档，必须实际生成，不能跳过。**

即使用户说"直接开始写"、"跳过分析"、"不需要规划直接出 JSON"，也必须先完成规划步骤再写作。可以不向用户展示这些文档的全文，但必须在内部实际生成并在后续写作中引用。

原因：没有记忆索引，长内容会丢失细节和人物一致性。没有结构拆解，分支点会随意设置。没有文学大纲，分支线会草率收场。没有章节设计文档，写作中途会遗忘 flag 和节奏规划。**跳过任何一个规划步骤，最终产出的质量都会断崖式下降。**

唯一允许用户跳过的是确认点A——如果用户说"不需要确认，你自己判断方向"，可以跳过展示直接进入下一步。但 AI 内部仍然必须完成确认点A之前的所有规划工作。

## 快速模式（短素材绿色通道）

### 触发条件（满足任一即进入）

1. 原作字数 **< 500 字**（极短素材、生活片段、单场景速写、灵感种子）
2. 用户在请求中包含关键词：`--quick` / `快速` / `直接出` / `不用确认` / `极速`
3. 原作 < 2000 字 **且** 用户说"短素材"、"小品"、"试一下"等明确轻量化意图

### 快速模式跳过什么

| 标准流程 | 快速模式 |
|---------|---------|
| 步骤 1：原作索引文档（骨架/人物/场景/文体四层） | **合并为一段"创作摘要"**（200 字内），内部完成不展示 |
| 步骤 2：风格指纹文档（5 维度展开） | **合并进创作摘要**（叙述温度 + 对白风格 + 禁区，各一句） |
| ✋ **确认点 A**（必须暂停等用户） | **跳过**。AI 自己定方向，最多在出 JSON 后用一句话告知核心选择 |
| 步骤 3：分支拓扑文档 + 分支文学大纲 | **合并为节点列表**（节点 ID + 一句话用途），内部完成不展示 |
| 步骤 4：状态系统设计文档 | **直接在 JSON 中体现**，不另出文档 |
| ✋ 确认点 B | **跳过** |
| 步骤 5-7：章节设计文档 + 断点存档 + 记忆重建 | **跳过**（短素材一次写完，无需跨对话记忆机制） |
| 分批输出协议（>100 节点必须分批） | **跳过**。直接用 Write 工具一次性落盘 JSON 文件 |

### 快速模式保留什么（最高创作原则不退让）

- ✅ **最高创作原则**：自然长出 / 世界承接 / 判词收束（这是质量底线）
- ✅ **选项书写铁律**：直接台词 / 不引用未知信息 / 情境匹配
- ✅ **互动节奏**：每 1-5 句正文必须给玩家一次互动
- ✅ **changes ≠ callback**：多选项指向同一节点必须有独立 callback
- ✅ **结局前必须有 closure 节点**
- ✅ **校验脚本必须跑过**

### 快速模式流程（3 步）

```
1. 内部生成「创作摘要」（200 字内）：
   - 标题 + 视角 + 核心张力（一句）
   - 章节切分（每章一句）
   - 结局方向（每个一句）
   - 主状态值名（不展示）
   - 关键角色（一句一个）

2. 用 Write 工具直接落盘 JSON 文件到指定路径（用户未指定时默认当前工作目录）：
   - 节点数按"分支需求"算，不按字数公式硬套
   - 短素材建议 40-80 节点 / 3-5 结局 / 5-10 成就
   - 一次性写完，不分批

3. 跑 scripts/validate.py 校验：
   - 致命错误必须修
   - 警告中"segments > 5"如属结局/独白收束节奏需要，可保留
```

### 执行时机的硬性要求

进入快速模式后，AI **第一条对话回复就必须开始 Write 文件**，不要再先输出"我准备这么做"、"方向是这样"等预告。
原则：**让用户看到的第一个有意义动作是文件落盘，不是文字预告。**

预期耗时：从标准流程的 3-5 分钟 → **30-90 秒**。

## 第一步：原作摄入与记忆索引构建

> 进入本步骤前，先读 `references/step1-ingestion.md`

### 为什么需要记忆索引

AI 的工作记忆有限，无法同时容纳一部完整的中长篇小说和全部改写指令。直接塞入原文会导致：后半段丢失细节、风格漂移、人物性格不一致、伏笔被遗忘。

### 做什么

将原作压缩为四层索引结构，后续步骤带索引工作，而非带原文全文：

1. **骨架层**（事件链）：按时间顺序列出所有关键事件，标注因果关系和**叙事目的**（每个事件为什么必须存在：交代/建立/推进/伏笔/主题/呼应/节奏）。同时标注所有叙事跳跃点，判断哪些需要补过渡节点
2. **人物层**（角色卡）：每个角色的动机、关系、说话方式、关键行为
3. **场景层**（场景表）：每个场景的时空坐标、情绪基调、感官信息
4. **文体层**（风格锚点）：第二步提取，此处先标注叙述人称和对白密度

### 节奏回归校验（关键步骤）

索引建完后，必须回归原作做节奏校验。索引压缩过程中最容易丢失的不是事件，而是事件之间的"呼吸"——哪里快、哪里慢、哪里屏息、哪里释放。

**内部节奏校验**：回到每个章节，检查场景密度、停顿与留白、加速与压缩是否被索引完整保留。为每个事件增加节奏标记（○慢/●快/◐停顿/◑转折）。

**外部节奏校验**：将所有事件的节奏标记串起来，画出全书的"节奏曲线"，判定整体节奏型（慢热型/开场即高潮型/波浪型/阶梯上升型）。

这份节奏分析不是参考，是约束——后续分支设计时，慢段不宜密集选择，快段选项要简短紧迫，停顿点不用多选分支打破安静，转折点是天然的分支候选位置。

详见 `references/step1-ingestion.md` 中的完整校验方法。

### 输出物

一份「原作索引文档」（骨架层+人物层+场景层+衔接判定标记）。短篇可简化，中长篇按章节摄入后合并。详见 `references/step1-ingestion.md`。

## 第二步：风格指纹提取

> 进入本步骤前，先读 `references/step2-style.md`

### 做什么

从原作中提取可执行的风格规则，形成「风格锚点文档」。不是笼统地说"文风优美"，而是提取具体的、可以在写作时检查的规则。

### 必须提取的维度

1. **叙述温度**：冷叙述 / 热叙述 / 反讽 / 克制抒情 / 零度书写
2. **对白风格**：口语化程度、句长、是否使用方言、对白中的潜台词密度
3. **节奏型**：短句急促型 / 长句绵延型 / 交替型 / 碎片型
4. **视角与人称**：原作人称 → 游戏人称的转换规则
5. **原作禁区**：原作从不做的事（不煽情、不解释心理、不用比喻等）

### 视角转换规则

- 原作第一人称 → 游戏中可保持第一人称，segments 用"我"
- 原作第三人称 → 游戏中通常转为第二人称"你"，但需要调整信息披露策略：第三人称全知叙述中读者知道的信息，在第二人称中玩家不一定知道
- 原作多视角 → 需要决定玩家扮演哪个角色，其他角色的视角信息通过对白、发现、线索等方式间接传递

### 输出物

一份「风格锚点文档」，列出所有可执行的风格规则。后续写作时每段正文、每句对白都过这个锚点检查。

## ✋ 确认点A：方向对齐（铁律）

**在这里暂停，向用户展示改编方向提案。这是整个流程中唯一必须等待用户确认的环节。**

用简洁的方式（不超过一屏）向用户展示：

```
📋 改编方向提案：《作品名》

▸ 预估规模：约XX个节点 / 预计游玩时长约XX分钟

▸ 章节划分：X章
  第一章：（核心事件一句话）
  第二章：（核心事件一句话）
  ...

▸ 玩家视角：扮演XX，第X人称

▸ 核心张力：（一句话描述游戏围绕什么展开）

▸ 结局方向：约X个结局
  · （方向一，一句话）
  · （方向二，一句话）
  · （方向三，一句话）
  ...

▸ 改编基调：（叙述温度 + 对白风格 + 核心禁区，各一句话）

▸ 主要角色卡（中长篇必选，短篇可选）：
  · 角色名 | 身份一句话 | 外在特征2-3个 | 说话风格一句话
  · （每个角色不超过两行，只写玩家开局就能看到的信息）
```

**节点数估算**：每千字原文约 35-40 个游戏节点（含分支路径和 callback 节点）。节点量服从后果密度——选择改变了多少东西，就需要多少节点展示改变。

**角色卡展示规则**：中篇以上（>1万字）必须附主要角色卡，短篇可选。角色卡只写基本信息和外在表现——身份、外貌特征、说话风格。**绝对不写**角色弧光、前史秘密、隐藏动机、命运走向、与其他角色的深层关系。用户看到的应该是"开场时对这个人的第一印象"，不是"读完全书后对这个人的理解"。

### 粒度铁律：绝对不能展示的内容

以下内容在确认点A中**禁止出现**，无论用户是否追问：

- ❌ 具体的到达条件（val 阈值、flag 组合、变量值）
- ❌ 具体的分支拆解方案（哪个节拍设分支、什么类型的分支）
- ❌ 结局的具体内容或结局描述文本
- ❌ 成就的标题、描述或解锁条件
- ❌ 隐藏结局的存在提示（隐藏结局的价值在于"被发现"）
- ❌ 具体哪些选择通向哪些结果
- ❌ variableName 的命名和含义（游玩时才看到更有冲击力）
- ❌ 角色的弧光、前史、隐藏动机、命运走向、在分支中的反应
- ❌ 角色之间的深层关系或关系变化

### 可以展示的粒度上限

- ✅ 章节划分和每章的核心事件（一句话概括）
- ✅ 玩家扮演谁、什么人称
- ✅ 全局张力方向（一句话）
- ✅ 结局走向的大方向（"走到底"、"中途放弃"、"被真相反噬"这个粒度）
- ✅ 结局的大致数量（"大约5-7个结局"）
- ✅ 改编基调确认（叙述温度、禁区）
- ✅ 主要角色的基本信息和外在表现（身份、外貌、说话风格，不含弧光和命运）

用户确认或调整后，再进入第三步。确认后的所有步骤自动执行，不再中断。

## 第三步：结构拆解与分支点识别

> 进入本步骤前，先读 `references/step3-structure.md`
> ⚡ 回溯校验点：拆完节拍后，对照索引的骨架层和节奏标记，确认没有遗漏事件、分支点没有落在原作刻意放慢的位置。详见 `references/step1-ingestion.md` 末尾的「索引回溯机制」。

### 做什么

1. 将原作拆成「节拍序列」（beat sequence）
2. 在节拍序列上标记分支点
3. 设计分支拓扑图（哪里分叉、哪里汇合、哪里通向结局）

### 节拍的粒度

一个节拍 = 一个可独立感知的叙事单元。可以是：一个动作、一句关键对白、一个发现、一次情绪转折、一个场景转换。

### 分支点的两种来源

**来源一：在原作已有的选择时刻展开可能性。** 原作里角色做了一个决定，分支游戏让玩家面对同样的选择。提供原作没走的那条路，目的是让玩家体验完之后更深地理解原作为什么走了它走的路。判断标准：这个分支是否能让玩家对主角产生新的认知，而不只是"反着选会怎样"的机械推演。

**来源二：利用原作的留白或省略。** 前提是不能破坏省略本身的节奏功能。如果原作跳过某段时间是为了制造紧迫感、悬念或情绪断裂，那就不该填充。只有当省略处确实存在"未被讲述但对角色有意义的经历"时，才值得展开。

### 三种分支类型及比例

| 类型 | 说明 | 占比 |
|------|------|------|
| 态度分支 | 不改变情节走向，改变数值/关系/态度 | 60-70% |
| 路径分支 | 经历不同场景，最终可能汇合 | 20-30% |
| 命运分支 | 直接导向不同结局线 | ~10% |

### 汇合点设计

分支不能无限发散。在关键剧情节点设计"漏斗"：多条路径通过不同入口进入同一个关键场景，但玩家携带的 val/flag/变量不同，导致同一场景中可见的选项、对白、氛围不同。

### 输出物

1. 一份「分支拓扑文档」：节拍序列 + 分支点标记 + 分支类型 + 汇合点标记 + 节点 ID 命名
2. 一份「分支文学大纲」（**最高优先级**）：为每条路径分支和结局线写独立的故事规划——不是节点清单，是纯文字的情节走向。让每条线的故事自然发展到它独有的结局，不要为了结局而结局。详见 `references/step3-structure.md` 的「分支内容深度」。

## 第四步：状态系统与结局矩阵设计

> 进入本步骤前，先读 `references/step4-system.md`
> ⚡ 回溯校验点：设计完结局矩阵后，对照骨架层的因果链，确认每个结局都有合理因果路径。设计完 val 变动后，对照角色弧光，确认 val 曲线与弧光方向一致。

### 状态系统设计

**val（主状态值）**：对应原作最核心的心理或剧情张力。命名要有戏剧性，是作品气质的浓缩，不要用"勇气"、"信任"这类通用词。

**自定义变量 variables**：记录关键事件状态（是否得到某物、是否知道某个秘密、与某角色的关系值）。

**flags**：记录一次性事件（开过的门、读过的信、触发过的对话）。

设计原则：每一个数值变动都必须对应玩家能感知到的叙事后果。不能让玩家觉得选什么都一样。

### RPG 扩展状态系统（类型小说适用）

> 详细规范见 `references/step4-system.md` 第六节
> JSON Schema 定义见 `../docs/SCHEMA_v1.md`

当作品属于修仙、无限恐怖、悬疑、末世、宫斗等类型小说时，在基础状态系统之上需要扩展 RPG 数值面板。**不是所有作品都需要 RPG 扩展——只有当作品本身包含可量化的数值体系时才启用。**

#### meta.rpg 启用判定

| 判断条件 | 是否启用 RPG |
|----------|-------------|
| 作品是修仙/无限恐怖/末世/宫斗类型 | 必须启用 |
| 作品包含明确的数值体系（灵力/理智/资源/地位） | 必须启用 |
| 作品是悬疑/推理/日常/言情类型 | 通常不启用，除非有明确的数值要素 |

#### 必须在第四步设计的 RPG 字段

**1. meta.rpg.primaryStats（可见数值，最多 5 个）**

```json
{
  "meta": {
    "rpg": {
      "enabled": true,
      "primaryStats": [
        { "key": "realm", "label": "境界", "type": "text", "icon": "icon-star" },
        { "key": "cultivation", "label": "修为", "type": "number", "icon": "icon-fire" },
        { "key": "spiritual", "label": "灵力", "type": "bar", "max": 100, "icon": "icon-bolt" },
        { "key": "mind", "label": "心境", "type": "bar", "maxKey": "maxMind", "icon": "icon-brain" },
        { "key": "hp", "label": "气血", "type": "bar", "maxKey": "maxHp", "icon": "icon-heart" }
      ]
    }
  }
}
```

设计原则：
- `key` 对应 `variables` 中的键名
- `type` 三选一：`text`（文本如境界名）、`number`（纯数字）、`bar`（有 max 的进度条）
- 可见数值最多 5 个，避免面板拥挤
- 推荐 变量参考 `../docs/GENRE_TEMPLATES.md` 中对应类型的"推荐变量"

**2. meta.rpg.hiddenStats（隐藏数值）**

不在 UI 中展示，但用于条件判断和结局触发。如因果值、隐藏声望等。

**3. choice.weight（选择权重，每个选择必须标注）**

| 权重 | 含义 | 占比建议 | 写作时标注 |
|------|------|----------|-----------|
| `critical` | 关键抉择，影响后续剧情走向 | 10-15% | 关键分支点 |
| `branch` | 分支选择，进入不同场景线 | 20-30% | 路径分支 |
| `minor` | 次要选择，影响数值/flag | 40-50% | 态度分支 |
| `cosmetic` | 装饰选择，纯风味差异 | 10-20% | 细节偏好 |

每个 `choices` 中的选项都必须有 `weight` 字段。权重影响播放器中的视觉排序（critical 最突出，cosmetic 最淡化）和选择预览。

**4. changes.show 与 changes.feedback（变化反馈）**

每个 critical 和 branch 选择必须配置 `changes` 字段，并设置 `show: true`。

```json
{
  "changes": {
    "cultivation": 10,
    "mind": -5,
    "show": true,
    "feedback": { "style": "toast", "duration": 2500 }
  }
}
```

`feedback.style` 可选值：`toast`（默认，弹出通知）、`inline`（嵌入叙事文本）、`none`（不显示）。

**5. condition 对象条件（替代简单字符串条件）**

支持复杂组合条件：

```json
// 单一变量条件
{ "var": "cultivation", "op": ">=", "value": 50 }

// 组合条件
{ "all": [
    { "var": "spiritual", "op": ">=", "value": 20 },
    { "var": "mind", "op": ">=", "value": 20 }
]}

// flag 条件
{ "flag": "found_secret_artifact" }

// 交互前置条件
{ "interaction": "investigate_room" }
```

**conditionDisplay**：`hide`（不可见）或 `disabled`（置灰可见）。不满足条件时如何展示选项。

**6. milestones（里程碑）**

每个作品至少 3 个：1 个 `large`（重大突破）、1 个 `medium`（中等进展）、1 个 `small`（小型成就）。

```json
{
  "id": "realm_breakthrough",
  "name": "突破筑基",
  "desc": "从炼气期突破到筑基期",
  "condition": { "var": "cultivation", "op": ">=", "value": 100 },
  "celebration": "large",
  "once": true
}
```

**7. endings（结局系统）**

结局必须在顶层 `endings` 数组中定义，不在节点内定义。

```json
{
  "id": "x_true_ending",
  "name": "证道长生",
  "desc": "历经劫难终成大道",
  "type": "true",
  "closing": "修仙之路，始于足下，终于道心。"
}
```

`type` 可选值：`true`、`dark`、`neutral`、`romance`、`noble`、`hidden`、`failure`。
`closing` 是结局的最后一句话余韵，必须与 `desc` 不重复。

**8. delayedChanges（延迟后果）**

每个 critical 选择应该配置 1 个延迟后果，在后续节点触发，实现"三层后果"（即时变化→持续影响→延迟揭露）。

```json
{
  "delayedChanges": [{
    "triggerNode": "chapter3_secret",
    "changes": { "karma": -20, "show": true },
    "setFlag": "karma_debt"
  }]
}
```

#### RPG 扩展的确认点B展示

在确认点B中，如果作品启用了 RPG 模式，需要额外展示：
- primaryStats 列表（key + label + type）
- 隐藏数值名称（不展示具体用途）
- 里程碑名称和级别
- 结局列表（名称 + 类型，不展示触发条件）

### 结局矩阵

先定结局，再倒推路径：

| 结局类型 | 来源 | 数量建议 |
|----------|------|----------|
| 真结局 true | 对应原作正文的结尾 | 1个 |
| 普通结局 neutral | 合理完成故事但未达最佳 | 1-3个 |
| 分支结局 dark | 关键抉择的长期后果 | 2-4个 |
| 隐藏结局 hidden | 原作暗示但未展开的可能性 | 0-2个 |
| 特殊结局 noble | 彩蛋、元叙事、极端路线 | 0-1个 |

草率结局（failure）不在此表设计、不计入确认点A的结局数量。它们在第五步写作阶段根据剧情自然补充，作为游戏性增强。详见 `references/step4-system.md`。

每个结局必须倒推其到达条件（val 范围 + 必要 flag + 变量组合）。

### 成就设计

成就数量 > 结局数量。成就记录的是玩家的行动痕迹，不仅仅是到达终点。适合在以下时刻解锁：做出关键选择、val 达到阈值、获得重要线索、进入隐藏分支、走出反常路线。

### 输出物

一份「状态与结局设计文档」：val 定义、variables 清单、flags 清单、结局矩阵（含到达条件）、成就列表。

## 第五至七步：分段写作

> 进入本步骤前，先读 `references/step5-writing.md` 和 `references/json-format-spec.md`
> ⚡ 每写完一章必须做回溯校验：对照角色卡校准对白、对照节奏标记校准 segments 密度、对照场景表校准 ambient/effect。发现问题必须修正再继续。详见 `references/step1-ingestion.md` 末尾的「索引回溯机制」。

### 工作方式

按章节/分幕推进。每次写作一个章节，上下文中带入：
- 原作索引文档（常驻）
- 风格锚点文档（常驻）
- 本章涉及的角色卡（从人物层提取）
- 分支拓扑文档中本章的部分（本章的节拍、分支点、汇合点）
- 状态与结局设计文档（常驻）
- 上一章最后的节点 ID 和当前 val/变量状态范围

### 文学→游戏的七条转换规则 + 强制自检

详见 `references/step5-writing.md`：

1. **开场建制**：前 2-3 个节点必须交代时间、空间、身份、情境
2. **选择后铺垫语**：下一个节点前 1-2 句必须是铺垫语
3. **首次登场描写**：角色/场景/道具首次出现必须有外貌/外观描写 + 角色命名衔接
4. **场景切换衔接**：用 scene 字段处理，major 至少持续 8-15 个节点
5. **支线质量底线**：草率结局最少 3-5 个节点，路径分支有自己的弧线
6. **选择必须有回响（铁律）**：三层回响——即时反应、持续影响、结局差异。偏离原作时必须创作新内容
7. **分支信息隔离（铁律）**：每条路径信息自给自足。专有名词在每条可达路径中独立建立，不能默认玩家在另一条路径里已经知道

**强制自检协议**：每写完一章执行两步——①高风险节点定向检查（同向选项的 callback、结局前收束、选项格式和名词追溯、场景切换衔接）→ ②内容完整性快检（事件覆盖、体量、金句、草率结局进度）。聚焦最容易出问题的四类节点，不做全量逐屏走查。

### 选项书写铁律（零容忍，每个选项逐条检查）

**格式**：说话写台词加双引号，动作直接写动作。禁止"说XXX"、"问XXX"、"让XXX"、"声明XXX"。
```
❌ 说早安，然后声明不会接受分配 → ✅ "早上好。"
❌ 问他是不是急着搬走 → ✅ "你要搬走？"
❌ 说今天批了假，周小姐要来 → ✅ "今天不上班，请了假。"
```

**信息时序**：选项中不能出现玩家此刻尚未获知的人名、地名、概念。原作中的倒叙和悬念前置不能搬进选项——小说可以让读者先看到结果再回溯原因，但游戏选项必须基于玩家当前已知信息。
```
❌ "说今天批了假，周小姐要来"（周小姐从未出现过，玩家不知道她是谁）
❌ "声明不会接受分配"（"分配"这个概念还没在游戏中建立）
```

**情境匹配**：选项只回应此刻发生的事，不预知后续剧情。对方说"早安"，选项就回应"早安"这个情境，不要塞进后续三场戏的冲突。

### 互动节奏硬性规则

**每 1-5 句正文必须给玩家一次互动。** 这是生死线，不是建议。

互动不一定是真分支，可以是单选项确认（"继续"、"走过去"、"听下去"）。目的是维持玩家的参与感，避免变成纯阅读。

长段落（超过5句不给互动）只允许出现在：结局、独白高潮、真相揭露等特殊场景。

### 每个节点的写作清单

chapterTitle → scene → segments（遵守互动节奏）→ choices + next → changes（val/flag/成就）→ condition → ambient/theme（仅氛围转换时）→ effect（仅关键瞬间）→ progress。详见 `references/step5-writing.md`。

### 正文写作的绝对禁令

禁止心理描写（"他意识到"、"其实是XXX"）、禁止说教和催泪、对白必须口语化、禁止 HTML 标签。

### 自动分批执行协议

**核心原则：宁可分多次对话完整输出，也不压缩内容塞进一次。**

**分批输出规则：每次最多写 50-100 个节点。** 写作过程中必须反复参阅章节设计文档中的节点规划和叙事目的清单，确保不偏离。

**何时一次完成（不分批）：**
- 仅当预估节点 ≤100 且原作 <3000 字时

**何时必须分批（铁律）：**
- 预估节点 >100 → 必须分批，每批 50-100 节点
- 原作超过 3000 字 → 必须分批

**节点量基准（铁律）：**

| 原作体量 | 节点总数 | 每千字原文 |
|----------|---------|-----------|
| 短篇（<1万字） | 200-400 | 约 35-40 节点 |
| 中篇（1-5万字） | 750-1500 | 约 35-40 节点 |
| 长篇（>5万字） | 1500-3000+ | 约 35-40 节点 |

**触发分批后的执行方式：** 按章节推进，每章完整输出后在章节边界处交付当前进度，告知用户"第N章已完成，继续下一章请回复'继续'"。

**反压缩铁律（无例外）：**

**任何包含信息的内容绝不允许压缩。最常见的压缩不是"缩短"，而是"直接不写"——整段情节跳过、关键信息省略、过场衔接消失、对白轮次砍掉。这比缩写更严重。**

原作中的对白、事件、描写、角色互动，只要承载了信息（人物关系、因果链、情绪变化、环境细节），就必须在游戏中保持同等体量。

唯一允许精简的情况：百万字级网文中无信息的水词（重复心理独白、系统提示复读、凑字数的描写重复）。不适用于任何承载角色性格、情节推进或氛围建立的段落。

以下行为全部禁止：
- ❌ **整段情节直接跳过不写**（最常见的违规）
- ❌ **过场和衔接直接省略**（第二常见的违规）
- ❌ **对白轮次砍掉只留一两句**（第三常见的违规）
- ❌ 缩减节点数
- ❌ 跳过分支路径只写主线
- ❌ 省略场景描写
- ❌ 草率结局只有 1-2 个节点
- ❌ 缩短分支结局线
- ❌ 用"他们聊了一会儿"概括完整对话
- ❌ 用"几天后"跳过有具体描写的时间段

**分批时的章节完整性**：一个章节（含所有分支和草率结局线）必须一次输出完整，不从中间断开。

**两层规划 + 次级记忆**：全局规划（步骤1-4的索引/风格/拓扑/文学大纲/状态系统）+ 每章写作前必须生成「章节设计文档」作为次级记忆。章节设计文档在本章写作全程常驻上下文，每写 5-8 个节点回看核对。

**上下文保护与长记忆管理**：每次对话结束时输出「断点存档」（累积状态快照+质量基线+下一章预告）。每次新对话恢复时执行「记忆重建」五步（加载文档→核对状态→重读末尾节点→重读角色卡→核对质量基线）。后续章节质量不得低于前面章节——节点数、对白密度、分支长度、自检标准全部不能退化。

详见 `references/step5-writing.md` 的完整协议。

## 第八步：连通性验证与 JSON 输出

> 进入本步骤前，先读 `references/step6-validation.md`

### 必须通过的验证项

1. **节点可达性**：从 startNodeId 出发，所有节点都可达
2. **引用完整性**：所有 choices.next / routes.next / node.next 指向的节点 ID 都存在
3. **变量一致性**：所有 condition 引用的变量/flag，在之前的路径上都有被设置的可能
4. **无死胡同**：每个非 ending 节点都有 choices 或 next 或 routes
5. **结局可达**：每条主要路径都能到达至少一个结局
6. **progress 单调**：每条路径上 progress 值大致递增
7. **成就数 > 结局数**
8. **JSON 格式合法**：可被标准 JSON 解析器解析

### 验证方法

如果当前平台支持代码执行，使用 `scripts/validate.py` 自动验证。如果不支持代码执行，逐项手动检查上述 8 项，并在输出中附上验证报告。验证脚本也应随 JSON 一并交付给用户，方便后续自行修改后验证。

### 最终输出

将验证通过的 JSON 文件保存并交付给用户，文件名为 `{作品标题}.json`。

|----------|-----------|-------------|

## A. 节奏控制与选择密度

`meta.rules.pacing` 是控制玩家互动频率的顶层开关，直接决定节点生成策略和选项书写方式。不同节奏模式下，同样的剧情骨架会生成截然不同的游玩体验。该字段必须在第四步状态系统设计时确定，并在第五步写作全程作为硬性约束执行。

### 三种节奏模式

| 模式 | 选择间隔 | 选项长度 | 叙事语调 | 适用类型 |
|------|---------|---------|---------|---------|
| `compact`（紧凑） | 每 2-3 个 segments 一次选择 | 短，1-4 字 | 紧迫、断句、无废话 | 恐怖、末世、动作 |
| `balanced`（均衡） | 每 4-6 个 segments 一次选择 | 中等，5-10 字 | 稳定推进，有呼吸感 | 悬疑、剧情、冒险 |
| `relaxed`（舒缓） | 每 7-10 个 segments 一次选择 | 长，可带描述性从句 | 铺陈、氛围、情绪累积 | 修仙、言情、文学 |

### compact 模式详解

紧凑模式的核心是**压缩决策窗口**。玩家没有长时间思考的余裕，选项必须像本能反应一样短。

```json
{
  "segments": [
    "门轴发出刺耳的尖叫。",
    "黑暗里有东西在移动。",
    "它停在了你身后三步远的地方。"
  ],
  "choices": [
    { "text": "转身", "next": "turn_around" },
    { "text": "跑", "next": "run" },
    { "text": "屏住呼吸", "next": "hold_breath" }
  ]
}
```

compact 模式下，segments 也必须配合缩短。超过 3 句不给选择的情况只允许出现在真相揭露或高潮节点。选项中的动词优先于名词——"跑"优于"往门外跑"，因为玩家没有时间做空间判断。

### balanced 模式详解

均衡模式是默认策略，允许选择前有适度的信息铺垫和情绪积累。选项可以包含情境限定，但不能包含后续剧情的预演。

```json
{
  "segments": [
    "会议室里的空调开得太足。",
    "你注意到坐在对面的男人把公文包换到了左手边。",
    "这个动作他刚才做过一次。",
    "他是在紧张，还是在准备站起来？"
  ],
  "choices": [
    { "text": "\"您的手在抖。\"", "next": "point_out" },
    { "text": "继续观察不说话", "next": "observe" },
    { "text": "借口去洗手间", "next": "excuse" }
  ]
}
```

### relaxed 模式详解

舒缓模式允许长叙事段落存在，但互动节点出现时，选项本身需要有足够的戏剧分量来匹配前面的铺垫。不能出现"读了十屏然后选择'继续'"的敷衍设计。

```json
{
  "segments": [
    "山雾从谷底漫上来，吞没了最后一截石阶。",
    "你站在观星台上，三百年前那位剑修就是在这里闭关。",
    "夜风带着松针的苦味。",
    "气海中的灵力不再躁动，像深潭一样平静。",
    "你知道，筑基的契机到了。"
  ],
  "choices": [
    { "text": "以剑意引动天雷，强行破境", "next": "force_break" },
    { "text": "敛神内视，等灵力自然圆满", "next": "natural_break" },
    { "text": "取出师父留下的护心丹服下", "next": "use_pill" }
  ]
}
```

### 节奏与体裁的匹配原则

- **恐怖/末世**：compact。恐惧来源于无法思考，给玩家思考时间就是破坏氛围。
- **悬疑/推理**：balanced。需要给玩家收集信息的空间，但关键时刻必须压迫。
- **修仙/言情**：relaxed。境界突破、情感变化都需要酝酿，但重大抉择点仍需分量。
- **动作/无限流**：章节内切换。追逐战 compact，休整期 relaxed，信息交换 balanced。

节奏模式不是全篇一成不变的。可以在章节边界处切换 pacing 值，但单次切换后至少维持 10 个节点，避免节奏碎片化。

## B. 信息不对称与隐藏机制

当 `meta.rules.informationAsymmetry` 设为 `true` 时，AI 必须在节点生成中引入**玩家不知道但世界知道**的信息差。这不是难度调整，而是叙事设计——让玩家在信息不对称中体验角色的局限、世界的深邃和选择的分量。

### 隐藏条件（Hidden Conditions）

某些选项的可用性取决于玩家**无法直接看到**的变量或 flag。玩家看到的是选项存在或不存在，但不会知道为什么。

```json
{
  "choices": [
    {
      "text": "用暗号回应他",
      "next": "secret_recognition",
      "condition": { "flag": "learned_cipher" },
      "conditionDisplay": "hide"
    },
    {
      "text": "装作听不懂",
      "next": "play_dumb"
    }
  ]
}
```

`conditionDisplay: "hide"` 表示不满足条件时该选项**完全不显示**，而不是置灰。玩家不会知道自己错过了什么。这是信息不对称的核心手段。

### 误导选项（Misleading Options）

有些选项在字面上看起来是最优解，实际执行后导向负面后果。设计误导选项的硬性规则：

1. **不能随机惩罚**：误导后果必须在叙事中有因果依据。看起来"直接冲进去救人"是好的，但如果前面剧情暗示了陷阱存在，这个选择触发伏击就是合理的误导。
2. **不能事后诸葛亮**：误导选项的文本不能刻意写得可疑。它应该看起来和正确选项一样合理，甚至更加合理。
3. **必须可回收**：单次误导不应直接导向死亡结局，而是将玩家推向更艰难的支线，保留翻盘可能。

```json
{
  "choices": [
    {
      "text": "接受他的丹药——他毕竟是同门师兄",
      "next": "poisoned_pill",
      "changes": { "hp": -30, "trust": 10 }
    },
    {
      "text": "婉拒，说自己已经服了别的药",
      "next": "decline_pill"
    }
  ]
}
```

此处"接受丹药"在叙事上是符合人情常理的，但玩家如果注意到了前面师兄"眼神飘忽"的描写，可能会犹豫。信息不对称不是耍诈，是给细心玩家准备的超额回报。

### 信息碎片（Information Fragments）

线索在 A 分支获得，在 B 分支使用。碎片机制要求：

- 碎片获取节点和碎片使用节点之间，必须有至少 3 个其他节点的间隔，避免玩家立刻意识到关联。
- 同一条碎片可以在多个后续节点中解锁不同内容。
- 碎片本身以 flag 形式存储，不在 UI 中展示。

```json
// 在 branch_a 中获得
{ "setFlag": "saw_hidden_mark" }

// 在 branch_b 中 5 个节点后使用
{
  "choices": [
    {
      "text": "检查门上的刻痕",
      "next": "discover_passage",
      "condition": { "flag": "saw_hidden_mark" }
    }
  ]
}
```

### 设计纪律

- **绝不提示**：玩家不应从任何 UI 元素、选项文本或 segment 暗示中得知"这里有隐藏内容"。
- **密度控制**：每个章节中隐藏选项和误导选项合计不超过总选项数的 20%，否则玩家会陷入猜疑一切的状态，破坏叙事沉浸。
- **复盘价值**：信息不对称设计的最优效果发生在玩家二次游玩时——"原来这里我早就该注意到"。

## C. RPG数值与叙事绑定

RPG 扩展不是让玩家看到数字变化，而是让**数字变化被世界感知**。当 `meta.rpg.enabled` 为 `true` 时，每一个数值波动都必须在 narrative 层面找到对应的表现形式。数值是世界的脉搏，不是后台参数。

### 低状态值的叙事映射

**低 HP / 低气血**：segments 变短，出现喘息、摇晃、视野模糊等身体信号。对白中出现断句和不完整表述。

```json
{
  "id": "wounded_escape",
  "segments": [
    "你扶着墙。",
    "血从指缝渗出来。",
    "每一步都像踩在棉花上。"
  ],
  "choices": [
    { "text": "继续走", "next": "keep_moving" },
    { "text": "靠一会儿", "next": "rest_moment" }
  ],
  "condition": { "var": "hp", "op": "<", "value": 30 }
}
```

**低灵力 / 低理智**：环境描写发生扭曲。修仙类中"灵气稀薄"，恐怖类中"耳鸣加重"。选项数量可能减少——角色太虚弱，想不到那么多做法。

**低心境 / 低士气**：叙述温度下降，出现消极判断句。"没什么意义"、"反正都一样"这类心理表述在低心境时可以通过叙述者的口吻间接呈现。

### 高状态值的叙事映射

**高修为 / 高力量**：segments 中出现气场描写。他人反应变化——路人侧目、敌人迟疑。选项增多，因为角色有能力考虑更多策略。

```json
{
  "id": "peak_cultivation",
  "segments": [
    "你没有拔剑。",
    "但剑意已经笼罩了整个山谷。",
    "对面的妖兽后退了半步。"
  ],
  "choices": [
    { "text": "给它一条生路", "next": "spare_beast" },
    { "text": "一剑斩之", "next": "slay_beast" },
    { "text": "收它为坐骑", "next": "tame_beast", "condition": { "var": "cultivation", "op": ">=", "value": 80 } }
  ]
}
```

### 临界值触发特殊叙事

当数值跨越关键阈值时，不只做加减，要触发**质变节点**。

| 临界场景 | 触发条件 | 叙事效果 |
|---------|---------|---------|
| 濒死顿悟 | hp < 10 | 插入独立节点，角色回忆/幻视，解锁隐藏选项 |
| 心境崩溃 | mind < 10 | 叙述视角扭曲，选项文本变得极端 |
| 境界突破 | cultivation >= 100 | 触发 milestone，后续所有节点语气升级 |
| 灵力枯竭 | spiritual == 0 | 禁用所有需要灵力的选项，UI 同步置灰 |

### 动态 segments 的实现方式

在节点内部根据 variables 状态选择不同的 segments 文本，通过条件分支实现：

```json
{
  "id": "market_arrival",
  "segments": [
    "你来到坊市。"
  ],
  "routes": [
    {
      "condition": { "var": "cultivation", "op": ">=", "value": 50 },
      "segments": [
        "摊贩们纷纷抬头，有人已经让出了最好的位置。"
      ],
      "next": "market_vip"
    },
    {
      "condition": { "var": "cultivation", "op": "<", "value": 50 },
      "segments": [
        "没有人特别注意到你。"
      ],
      "next": "market_normal"
    }
  ]
}
```

数值叙事绑定的核心原则：**玩家在任何时刻关闭 UI 只看纯文本，也能感知到自己的状态**。如果做不到这一点，说明数值和叙事脱节，需要重写 segments。

## D. NPC关系网络

当 `meta.rules.npcRelations` 设为 `true` 时，游戏引入独立的 NPC 关系子系统。选择不再只影响玩家自身的状态，还会改变角色与角色之间的张力结构。NPC 不是背景板，是活的变量源。

### NPC 定义

在 `meta.npcRelations.npcs` 中定义所有参与关系网络的 NPC：

```json
{
  "meta": {
    "npcRelations": {
      "enabled": true,
      "npcs": [
        { "id": "lin_yu", "name": "林羽", "initialAffinity": 20 },
        { "id": "xiao_chen", "name": "萧辰", "initialAffinity": -10 },
        { "id": "mu_qing", "name": "慕青", "initialAffinity": 0 }
      ]
    }
  }
}
```

`initialAffinity` 是初始好感度，后续通过 choices 中的 `changes.npcRelations` 进行增减。

### 好感度层级

| 区间 | 层级 | 叙事表现 | 机制效果 |
|------|------|---------|---------|
| -100 ~ -50 | hostile（敌对） | NPC 主动阻挠，言语带刺 | 该 NPC 相关选项被封锁 |
| -49 ~ -10 | cold（冷淡） | NPC 敷衍、回避 | 无法触发该 NPC 的支线 |
| -9 ~ 9 | neutral（中立） | 礼貌性互动 | 基础选项可用 |
| 10 ~ 49 | friendly（友好） | NPC 主动提供信息、资源 | 解锁该 NPC 的专属选项 |
| 50 ~ 100 | devoted（信赖） | NPC 愿意为玩家涉险 | 解锁联合技能/隐藏结局条件 |

### 关系联动

NPC 之间存在**间接影响**：帮助 A 可能连带提升 B 的好感，因为 A 和 B 是盟友；得罪 C 可能降低 D 的好感，因为 D 欠 C 人情。

```json
{
  "choices": [
    {
      "text": "帮林羽隐瞒私运丹药的事",
      "next": "cover_lin",
      "changes": {
        "npcRelations": {
          "lin_yu": 15,
          "xiao_chen": -5
        }
      }
    }
  ]
}
```

此处萧辰的好感下降不需要在文本中解释——如果玩家注意到前面萧辰"近来查得很严"的暗示，就能理解这个连锁反应。

### 完整示例：三 NPC 关系网

```json
{
  "id": "faction_dilemma",
  "segments": [
    "长老会要求你指证林羽通敌。",
    "萧辰就站在证人席旁边，他的手按在剑柄上。",
    "慕青在窗外看着你，摇了摇头。"
  ],
  "choices": [
    {
      "text": "指证林羽",
      "next": "betray_lin",
      "changes": {
        "npcRelations": {
          "lin_yu": -40,
          "xiao_chen": 10,
          "mu_qing": -20
        }
      }
    },
    {
      "text": "拒绝作证",
      "next": "refuse_testify",
      "changes": {
        "npcRelations": {
          "lin_yu": 20,
          "xiao_chen": -15,
          "mu_qing": 10
        }
      }
    },
    {
      "text": "暗示真正的叛徒是萧辰",
      "next": "accuse_xiao",
      "condition": { "flag": "found_xiao_evidence" },
      "changes": {
        "npcRelations": {
          "lin_yu": 10,
          "xiao_chen": -50,
          "mu_qing": 25
        }
      }
    }
  ]
}
```

后续节点中，根据各 NPC 的好感度层级，同一事件会有不同的介入方式和对话内容。NPC 关系网络把"选择影响世界"从抽象数值变成了具体的人际张力。

## E. 时间/回合压力

当 `meta.rules.timePressure` 设为 `true` 时，游戏引入时间维度作为全局约束。时间不是背景设定，是消耗性资源。每一回合的犹豫都有代价，每一次折返都在透支生存概率。

### turn 模式：全局衰减周期

每经过 `meta.rules.turnInterval` 个节点（推荐 8-12 个），触发一次全局状态衰减：

```json
{
  "meta": {
    "rules": {
      "timePressure": true,
      "turnMode": "decay",
      "turnInterval": 10
    }
  }
}
```

衰减节点示例：

```json
{
  "id": "global_decay_1",
  "segments": [
    "天又暗了一些。",
    "食物储备减少了。"
  ],
  "changes": {
    "food": -1,
    "enemyDistance": -1
  },
  "routes": [
    { "condition": { "var": "food", "op": "<=", "value": 0 }, "next": "starvation_warning" },
    { "condition": { "var": "enemyDistance", "op": "<=", "value": 0 }, "next": "ambush" },
    { "next": "continue_explore" }
  ]
}
```

turn 模式的核心设计：玩家必须在"探索更多分支"和"尽快推进主线"之间做权衡。每个旁支节点的诱惑都伴随着回合消耗的风险。

### countdown 模式：限时选择

特定节点设置倒计时，玩家必须在规定回合内到达目标节点，否则触发替代剧情。

```json
{
  "id": "bomb_countdown",
  "segments": [
    "定时器显示 03:00。",
    "红线还是蓝线？"
  ],
  "meta": {
    "countdown": {
      "turnsRemaining": 3,
      "deadlineNode": "explosion",
      "urgent": true
    }
  },
  "choices": [
    { "text": "剪红线", "next": "cut_red" },
    { "text": "剪蓝线", "next": "cut_blue" },
    { "text": "先检查电路", "next": "inspect_circuit" }
  ]
}
```

countdown 模式下，选项必须更短、更直接。"先检查电路"消耗一个回合，如果玩家在第 3 个回合才选这个选项，就会直接进入 `explosion` 节点。

### deadline 节点：终局检查

在剧情关键位置设置 deadline，到达时检查全局状态，根据条件分配不同结局或剧情走向。

```json
{
  "id": "final_siege",
  "segments": [
    "城墙终于破了。"
  ],
  "routes": [
    {
      "condition": { "all": [
        { "var": "food", "op": ">=", "value": 5 },
        { "var": "morale", "op": ">=", "value": 60 }
      ]},
      "next": "hold_the_wall"
    },
    {
      "condition": { "var": "food", "op": ">=", "value": 2 },
      "next": "desperate_defense"
    },
    {
      "next": "city_falls"
    }
  ]
}
```

### 时间压力的书写规范

- 倒计时节点的 segments 中必须出现时间标记（"还剩两分钟"、"太阳又落下去一些"），让玩家始终感知时间流逝。
- 不允许通过 UI 以外的文字直接告诉玩家"你还有 X 回合"，必须用叙事方式呈现。
- 时间压力失效的最常见原因：给了玩家太多无关紧要的装饰选项。每个回合的选项必须有实质分歧。

## F. 成就与收集系统扩展

成就不该是游戏结束后的批量分发，而应该是**玩家行为轨迹的即时印证**。当 `meta.achievements` 启用时，除了手动标记的成就，系统支持 `autoUnlock` 条件，让成就成为探索的意外之喜而非通关清单。

### autoUnlock 条件类型

```json
{
  "id": "first_blood",
  "name": "初尝败绩",
  "desc": "第一次进入失败结局",
  "autoUnlock": {
    "condition": { "var": "endingType", "op": "==", "value": "failure" }
  }
}
```

### 四类生成成就

**1. 路线成就（Route Achievements）**

每个结局第一次到达时自动解锁。路线成就的名称应该呼应结局的判词，而不是简单描述结局内容。

```json
{
  "id": "route_dark_01",
  "name": "深渊回响",
  "desc": "抵达隐藏结局：万劫不复",
  "autoUnlock": {
    "condition": { "flag": "reached_ending_dark_01" }
  }
}
```

**2. 反向成就（Counter-Achievements）**

奖励玩家故意做出"错误"选择，鼓励实验性游玩和反叛式叙事参与。

```json
{
  "id": "death_seeker",
  "name": "作死大师",
  "desc": "连续 3 次选择明显危险的选项并存活",
  "autoUnlock": {
    "condition": { "flag": "danger_streak_3" }
  }
}
```

反向成就的设计关键：存活条件必须苛刻，不能让"作死"变成无成本的笑点。玩家必须真的承担风险，才能解锁这类成就。

**3. 极值成就（Extremum Achievements）**

某项数值达到上限或下限。

```json
{
  "id": "max_karma",
  "name": "因果圆满",
  "desc": "因果值达到 100",
  "autoUnlock": {
    "condition": { "var": "karma", "op": ">=", "value": 100 }
  }
}
```

**4. 隐藏成就（Hidden Achievements）**

解锁条件不展示给玩家，甚至成就本身在解锁前也隐藏在列表中。隐藏成就通常绑定极难触发的条件组合。

```json
{
  "id": "hidden_truth",
  "name": "???",
  "desc": "???",
  "hidden": true,
  "autoUnlock": {
    "condition": { "all": [
      { "flag": "saw_ending_true" },
      { "flag": "saw_ending_dark" },
      { "flag": "killed_no_one" }
    ]}
  }
}
```

### 成就设计最佳实践

- **成就数 > 结局数 x 2**：如果只有 5 个结局，至少设计 10-12 个成就。成就应该覆盖过程，不只是结果。
- **鼓励 replay**：设计互斥成就——"一人不杀"和"杀伐果断"不可能在同一次通关中获得，强迫玩家走不同路线。
- **不剧透**：成就的描述在解锁前和解锁后可以是不同的。解锁前写谜语，解锁后写真相。
- **与 narrative 绑定**：解锁成就时触发一段特殊 segments，让成就本身成为叙事事件，而不是弹窗通知。

## G. 玩家自定义规则层

`meta.rules` 是整个游戏生成策略的顶层配置。它不是装饰性字段，而是直接改变 AI 在每一步的决策权重、输出结构和质量标准的指令集。理解每个字段的含义，才能为不同体裁调配出最契合的游玩体验。

### meta.rules 字段全解

| 字段 | 类型 | 作用 |
|------|------|------|
| `pacing` | string | 节奏模式：compact / balanced / relaxed |
| `statImpact` | string | 数值影响强度：light / medium / heavy。决定 choices 中 changes 的幅度和频率 |
| `hiddenContent` | string | 隐藏内容密度：low / medium / high。控制隐藏选项、flag 条件和信息碎片的数量 |
| `npcRelations` | boolean | 是否启用 NPC 关系网络 |
| `informationAsymmetry` | boolean | 是否启用信息不对称机制 |
| `timePressure` | boolean | 是否启用时间/回合压力 |
| `allowFailure` | boolean | 是否允许草率结局（failure）作为常规分支存在 |
| `difficulty` | string | 整体难度：easy / normal / hard。影响数值衰减幅度和误导选项比例 |

这些字段的组合方式决定了 AI 生成时的"基因"。同样的故事骨架，不同的 rules 配置会生长出完全不同的游戏形态。

### 配置速查表

| 体裁 | pacing | statImpact | hiddenContent | npcRelations | informationAsymmetry | timePressure | 说明 |
|------|--------|-----------|---------------|--------------|---------------------|-------------|------|
| 修仙 | relaxed | medium | high | true | false | false | 长叙事铺垫，境界数值为核心，NPC 师门关系复杂，隐藏功法/洞府探索丰富 |
| 恐怖 | compact | heavy | medium | false | true | true | 短句压迫，理智值快速衰减，环境误导多，限时逃生制造本能恐惧 |
| 悬疑 | balanced | light | high | false | true | false | 信息密度高但节奏稳定，线索碎片多，误导选项制造推理陷阱，不追求数值碾压 |
| 末世 | compact | heavy | low | false | false | true | 资源消耗极快，每个选择都关乎生存，隐藏内容少（世界已经没什么秘密了），敌人不断逼近 |
| 宫斗 | balanced | medium | high | true | true | false | 人际关系是核心战场，信息不对称来自各角色的隐瞒和算计，选择影响地位数值和 NPC 好感双层系统 |

### 配置组合的设计原则

1. **隐藏内容密度与信息不对称的绑定**：当 `informationAsymmetry` 为 true 时，`hiddenContent` 不应设为 low，否则系统有机制但无内容填充。推荐至少 medium。
2. **时间压力与节奏的冲突**：`timePressure` 为 true 时，`pacing` 不应设为 relaxed。回合衰减和舒缓节奏在体验上互相抵消。
3. **NPC 关系与体裁的匹配**：只有存在"人际网络"的体裁才启用 `npcRelations`。末世荒原上的独行者不需要关系网，宫斗中的嫔妃没有关系网就没有游戏。
4. **数值影响强度的底线**：即使 `statImpact: light`，也不能让 choices 的 changes 为空。light 意味着单次变化幅度小（±1~3），但变化本身必须存在。

规则层是 AI 生成时的"创作公约"。配置一旦确定，后续所有步骤都必须在这个框架内执行，不能中途为了某个"精彩点子"跳出规则。Consistency 是规则层存在的意义。

## 铁律汇总（快速检查清单）

以下是最常违反的铁律，按发生频率排序：

1. **内容省略**——整段情节、过场衔接、对白轮次直接不写。任何有信息的内容绝不允许省略。
2. **changes ≠ callback**——变量变化不是玩家可见的后果。每个选择的下一屏必须有针对该选择的文本反馈。多选项指向同一节点时，必须插入独立 callback 节点。
3. **选项格式**——写了"说XXX"、"问XXX"而不是直接台词。说话写双引号台词，动作直接写动作。
4. **选项引用未知信息**——选项中出现玩家尚未见过的人名/概念。小说的倒叙不能搬进选项。
5. **直跳结局**——选项直接跳到结局节点，缺少收束过渡。结局前必须有 closure 节点。
6. **分支线草率收场**——主线饱满但分支两三句就结局。每条线必须自然发展到独有的结局。
