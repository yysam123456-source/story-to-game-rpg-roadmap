# 第三步：结构拆解与分支点识别

## 目标

将线性叙事拆成节拍序列，在节拍序列上识别分支点，设计分支拓扑图。

## 前置依赖

本步骤必须在完成第一步（含节奏回归校验）之后进行。第一步生成的**节奏标记**（○慢/●快/◐停顿/◑转折）和**全书节奏曲线**是分支设计的刚性约束：

- 标记为 `○ 慢` 的区域：不宜密集插入多选分支，以免打破铺垫的沉浸感
- 标记为 `● 快` 的区域：选项要短、紧迫，segments 保持高密度
- 标记为 `◐ 停顿` 的区域：可用单选确认维持参与感，不用多选分支破坏安静
- 标记为 `◑ 转折` 的位置：天然的分支候选点——节奏突变往往也是角色面临选择的时刻

在下面的分支点识别阶段，每标记一个分支点时都必须回查它所在区域的节奏标记，确认分支设计与原作节奏不冲突。

## 阶段一：节拍拆解

### 什么是节拍

一个节拍（beat）是叙事中最小的可感知单元。它可以是：

- 一个动作（他推开门）
- 一句关键对白（"你不该来这里"）
- 一个发现（桌上有一封拆开的信）
- 一次情绪转折（气氛从轻松变为紧张）
- 一个场景转换（从室内到室外）
- 一个时间跳跃（三天后）
- 一个信息揭露（原来他早就知道）

### 拆解粒度

不要拆得太细（每句话一个节拍），也不要拆得太粗（一整章一个节拍）。

参考标准：**如果删掉这个节拍，读者是否会感到困惑或遗漏？** 如果是，它就是一个独立节拍。如果删掉后无感，它应该合并到相邻节拍中。

### 节拍序列格式

```
B01 [场景/动作] 主角在走廊尽头的门缝里发现一封信 → B02
B02 [选择时刻] 主角决定是否拆开信 → B03a / B03b ★分支点
B03a [动作/发现] 主角拆开信，读到内容 → B04
B03b [动作] 主角把信放回去 → 草率结局线
B04 [对白/信息] 信上写着"如果你还记得海" → B05
B05 [选择时刻] 主角决定是否去天台 → B06a / B06b ★分支点
```

标记说明：
- `★分支点` = 在此处设计玩家选择
- `→` = 因果流向
- `B03a / B03b` = 分支后的不同路径

## 阶段二：分支点识别

### 识别标准

一个节拍是否应该成为分支点，问三个问题：

**1. 这里是否存在真实的选择张力？**
角色在此处是否面临两种以上合理的行动方向？如果只有一种合理选择（比如"逃跑"在面对火灾时），这不是好的分支点。

**2. 不同选择是否导致有意义的差异？**
选A和选B之后，叙事体验是否真的不同？如果两条路在两步之后就汇合到完全相同的内容，这个分支点就是假的。

**3. 玩家是否有足够的信息做出这个选择？**
如果选择的后果完全不可预测（选哪个都是盲猜），玩家会觉得被欺骗。好的选择是：玩家能大致预感到方向，但不确定具体后果。

### 分支点的两种来源

**来源一：原作已有的选择时刻**

原作中角色做了一个决定的地方。分支游戏提供原作没走的那条路。

判断标准：这个分支是否能让玩家对主角产生新的认知？

好的例子：原作中主角选择原谅了对手。分支游戏让玩家可以选择不原谅——走了不原谅的线后，玩家发现不原谅的代价很重，从而理解了原作中原谅的重量。

坏的例子：原作中主角走了左边的路。分支游戏让玩家可以走右边——但走右边只是换了一段等量的描写，到达同一个目的地。这是无意义的分支。

**来源二：原作的留白或省略**

前提：不能破坏省略的节奏功能。

判断流程：
1. 原作为什么跳过了这段？
2. 如果跳过是为了节奏（压缩时间、制造紧迫感、留悬念）→ 不展开
3. 如果跳过的部分确实包含"未被讲述但对角色有意义的经历"→ 可以展开为支线
4. 展开的支线不能改变主线走向，只能增加对角色的理解深度

### 三种分支类型

**态度分支（60-70%）**
- 不改变情节走向
- 改变的是：val 数值、角色关系变量、玩家获得的信息、NPC 对玩家的态度
- 适用场景：对话中的回应方式、观察时关注什么、面对冲突的态度
- 特点：同一个 next 目标，但 changes 不同

示例：
```json
{
  "choices": [
    {
      "text": "直视他的眼睛",
      "next": "confrontation_02",
      "changes": { "val": 8, "set": { "stance": "bold" } }
    },
    {
      "text": "低下头不说话",
      "next": "confrontation_02",
      "changes": { "val": -5, "set": { "stance": "passive" } }
    }
  ]
}
```

**路径分支（20-30%）**
- 玩家经历不同的场景或事件序列
- 最终在某个汇合点重新合流
- 但汇合时，玩家携带的 val/flag/变量不同，导致后续选项和对白有差异
- 适用场景：选择不同的行动方案、去不同的地点调查、和不同的人交谈

示例：分支出两条路径，各有 3-5 个节点，最终汇合到同一个关键场景。

**命运分支（~10%）**
- 选择直接导向不同的结局线
- 不再汇合
- 适用场景：故事的重大转折点、不可挽回的决定
- 通常出现在故事的中后段

## 阶段三：汇合点设计

### 为什么需要汇合

如果每个分支都永远不汇合，内容量会指数爆炸。一个 5 层的二叉分支树有 32 个叶节点——每个都要写完整的后续内容，不现实。

### 汇合的方法

**方法一：硬汇合**
多条路径在某个事件节点直接汇合。叙事上的合理性来自：这个事件无论之前走了哪条路都会发生（比如"天亮了"、"电话响了"、"有人敲门"）。

**方法二：软汇合**
多条路径进入同一个场景，但场景内部根据 flag/变量/val 展示不同的 segments 和选项。同一个节点 ID，但玩家的体验因携带的状态不同而不同。

```json
{
  "choices": [
    {
      "text": "把线索交给他",
      "next": "reveal_truth",
      "condition": "hasFlag 'found_letter'"
    },
    {
      "text": "承认自己什么都不知道",
      "next": "reveal_truth",
      "condition": "!hasFlag 'found_letter'"
    }
  ]
}
```

**方法三：漏斗收束**
用 routes 自动路由，根据 val 范围将多条路径收入少数几条后续线。

```json
{
  "routes": [
    { "condition": "val >= 60", "next": "path_trust" },
    { "condition": "val >= 30", "next": "path_neutral" },
    { "condition": "default", "next": "path_suspicion" }
  ]
}
```

### 汇合点的位置

建议在以下叙事节点设置汇合：
- 章节/幕的交界处
- 不可避免的剧情事件（无论如何都会发生的事）
- 角色的被动遭遇（被找到、被告知、被卷入）

## 过渡节点规划

对照第一步事件链中每个衔接判定标记（`[空间过渡]`、`[时间过渡]`、`[情绪过渡]`、`[信息桥接]`），将标记为"需补"的衔接转化为实际的过渡节点，纳入节拍序列。

### 过渡节点的四种类型

**1. 移动过渡**（对应 `[空间过渡:需补]`）
补 1-2 个节点写移动过程。用 transient 场景标记中间地点。

```
事件链标记：[空间过渡:需补] 房间→天台，计划2个过渡节点
节拍序列：E04a(房间) → T01(走廊,transient) → T02(楼梯,transient) → E05(天台,major)
```

**2. 时间过渡**（对应 `[时间过渡:需补]` 或 `[时间过渡:轻描]`）
- 需补：补 1-3 个节点交代跳过的时间里发生了什么
- 轻描：在前一个节点末尾加一句或用 scene.arrival 处理

**3. 情绪过渡**（对应 `[情绪过渡:需补]`）
补一个缓冲节点：安静的动作（走到窗边、洗把脸）、一句环境描写（雨停了）。不写心理描写。

**4. 信息桥接**（对应 `[信息桥接:需补]`）
补一个信息获取节点：一通电话、一张纸条、某人的一句话。

### 过渡的分寸与节奏保护

**不要过度补。** 过渡节点是胶水，不是新章节。加过渡的目的是维持沉浸感，但加太多反而会拖慢节奏、稀释叙事密度。

**必须对照节奏标记决定过渡方式：**

| 当前节奏区域 | 过渡策略 |
|-------------|---------|
| ● 快节奏段 | **最轻量处理**。空间过渡用 scene.arrival 一句话带过，不加独立过渡节点。时间过渡用一句 segments 跳过。绝不在追逐/争吵/高潮段塞过渡节点拖节奏 |
| ○ 慢节奏段 | 可以补独立过渡节点（1-2个），让玩家有呼吸空间。移动过渡可以写稍细致的环境观察 |
| ◐ 停顿段 | 过渡本身就是停顿的一部分，可以用安静的单选确认节点实现。不需要塞事件 |
| ◑ 转折段 | 转折前可以补一个短过渡做铺垫（"暴风雨前的安静"），但转折本身不能被过渡稀释 |

**三条红线：**
1. 连续两个过渡节点之间不能没有剧情事件——如果出现"过渡→过渡→过渡"，说明补过头了
2. 快节奏段（标记为 ●）里不加独立过渡节点，全部用 scene.arrival 或 segments 内嵌处理
3. 过渡节点不应该包含选择分支——过渡是连接，不是岔路。如果过渡中出现了有意义的选择，它就不再是过渡，而是一个正式的剧情节点

标记为 `[自然衔接]` 的事件对不需要任何处理。标记为"轻描"的用 scene.arrival 或一句 segments 带过。只有"需补"的才规划独立过渡节点，且必须对照节奏标记决定轻重。

## 输出：分支拓扑文档

最终输出格式：

```markdown
# 分支拓扑：《作品名》

## 章节划分
（列出全书分为哪几个章节/幕）

## 节拍序列（含分支标记）
（完整的节拍序列，标注分支点、分支类型、汇合点）

## 分支拓扑图（文字描述）
（描述主线 + 各分支的走向 + 汇合位置 + 结局指向）

## 节点 ID 命名方案
（为所有节点预设 ID，格式：章节_场景_序号，如 ch1_letter_01）
```

## 分支内容深度（最高优先级铁律）

**这是整个技能中最重要的一条规则。**

**每一条分支线、每一个结局，都必须像原作主线一样充分、一样有探索性。不要为了结局而结局——让故事自然发展，直到它自己走到终点才停下来。**

### 问题的本质

AI 改编最致命的倾向是：原作主线写得饱满充实，但分支路径——尤其是偏离原作的路径——急于收场。"玩家选了不同的路→两三句交代→草草结局"。玩家一偏离主线就觉得"这条路没有被认真对待"，游戏的分支系统沦为装饰。

### 解决方案：先做文学规划，再写 JSON

**在第三步完成分支拓扑设计之后、第五步写 JSON 之前，必须为每条路径分支和结局线单独写一份文学大纲。**

这份大纲不是节点清单，是故事规划——用纯文字描述这条路上会发生什么、角色会怎样变化、局面会怎样发展、最终会自然走向什么样的结局。

**文学大纲的写法：**

```markdown
## 分支线：拒绝帮助老邱（从 ch2_refuse 分出）

### 故事走向
主角拒绝了老邱之后，老邱独自去了厂医办公室。主角以为事情跟自己无关了，
但第二天上班时发现老邱没来。去问同事，同事说老邱昨晚在厂里出了事。
主角开始后悔，决定自己去厂医办公室看看老邱到底想找什么。
但厂医办公室已经被锁了，门上贴了封条。
主角想办法从后窗进去（这里可以设一个分支——是冒险进去还是去找钥匙）。
在办公室里发现了老邱藏的那份文件，但同时听到有人来了……
（继续发展）……
最终主角拿到了文件但被发现，不得不选择是交出文件保全自己，
还是带着文件逃跑。无论哪个选择，都通向这条线独有的结局——
一个和真结局完全不同的、属于"拒绝帮助后自己卷入"的故事终点。

### 预估节点数：15-25
### 这条线独有的场景：被封的厂医办公室（夜晚版）、厂区后墙
### 这条线独有的角色互动：与看守的对话
### 结局方向：独立结局——"你拿到了真相，但再也没办法假装不知道。"

### 氛围规划
- 前半段（后悔、调查）：theme 保持主线的 noir，ambient 从 none 切到 static
- 进入封锁办公室：ambient 切 static，第一次发现文件时用 effect: flash
- 被发现时：effect: shake，紧接 pulse
- 结局：ambient 切 silence，closing 用 breath

### 关键选择设计
- "冒险从后窗进去" vs "去找钥匙"：两个都合理——冒险更快但有风险，找钥匙更安全但可能暴露意图。这个选择揭示玩家对风险的态度。
- "交出文件" vs "带着文件逃跑"：这是命运分支，通向两个不同的结局变体。交出文件 = 安全但真相被压下；逃跑 = 真相保住但自己陷入危险。

### val 变化规划
- 拒绝帮助老邱时 val -15（这个决定的代价从一开始就体现在数值上）
- 决定自己去调查时 val +10（开始弥补）
- 发现文件时 val +8
- 交出文件 val -20 / 逃跑 val +5
```

**关键原则：让故事自然生长到结局。**

不是"先定好结局是什么，然后凑几个节点通向它"。而是"从分支点出发，问自己：如果角色真的做了这个选择，接下来会自然发生什么？然后呢？再然后呢？"一直追问下去，直到故事自然走到一个终点——那个终点就是这条线的结局。

如果追问了三步就觉得"没东西可写了"，说明分支的起点选得不好，或者想象力不够。一个真正有意义的选择，它的后果至少可以延伸出十几个节点的故事。

### 文学大纲的完整要素

文学大纲不只是故事梗概——它是整条线的完整体验设计，包含叙事、交互和氛围三个层面：

**叙事层（必做）：**
1. 故事走向（至少 5-8 句描述情节发展，有起承转合）
2. 独有场景（这条线会经过哪些主线不经过的地点）
3. 独有角色互动（这条线会和哪些角色产生主线不会有的交集）
4. 原作金句在这条线中的位置（如果这条线经过某个金句所在的场景）
5. 结局方向（故事自然走到的终点）

**交互层（必做）：**
6. 关键选择设计（这条线上有哪些分支点，每个选择揭示玩家的什么态度/判断，为什么两个选项都合理）
7. val 变化规划（这条线上的主要 val 变动节点和变动方向，val 的变化曲线是否有叙事意义）
8. flag/变量规划（这条线会设置哪些独有的 flag，后续哪里引用）
9. 预估节点数（非草率结局不低于主线的 50%）

**氛围层（建议）：**
10. theme/ambient 的切换节点（在哪里换氛围、换成什么、为什么）
11. effect 使用规划（在哪些关键瞬间用什么特效）
12. scene 规划（哪些是 major 场景、哪些是 transient、大致的场景持续节点数）

### 更新后的输出要求

第三步的输出除了分支拓扑文档之外，必须增加：

```markdown
## 分支文学大纲

### 主线（真结局路径）
（完整的叙事+交互+氛围三层规划）

### 分支线1：XXX（从XX处分出）
（完整的三层规划）

### 分支线2：XXX（从XX处分出）
（同上）

...每条路径分支和结局线各一份
```

这份文学大纲在第四步设计状态系统和第五步写 JSON 时都必须带入上下文。第四步从大纲中提取 val/变量/flag 的全局设计；第五步逐章对照大纲写作。
