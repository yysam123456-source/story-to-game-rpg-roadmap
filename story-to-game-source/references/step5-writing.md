# 第五至七步：分段写作

## 目标

将前四步的设计成果转化为实际的 JSON 节点内容。按章节/分幕推进，每次写一个章节。

## 写作环境准备

每次写一个章节时，上下文中必须包含：

1. **原作索引文档**（常驻）
2. **风格锚点文档**（常驻）
3. **本章涉及的角色卡**（从人物层提取相关角色）
4. **分支拓扑文档的本章部分**（本章的节拍序列、分支点、汇合点）
5. **状态与结局设计文档**（常驻）
6. **上一章末尾状态**：最后的节点 ID、当前 val 范围、已设置的 flags 和变量
7. **原作对应章节的原文**（如果上下文容得下的话；容不下则依赖索引）

## 核心创作规则（七条转换规则 + 选项铁律 + 逻辑连贯性）

以下规则解决文学文本转游戏时最容易出现的体验断裂。

### 规则一：开场建制

很多小说/剧本为了文学语言的考虑，开场不做建制——可能直接进入一段氛围描写、一句对白、一个动作。这在阅读中没问题，读者会跟着文字慢慢弄明白背景。但游戏中，玩家需要先回答三个问题才愿意投入：**我是谁、我在哪、这是什么时候的事。**

**执行规则：**

游戏的前 2-3 个节点必须完成建制，即使原作没有明确的建制段落。建制不是写说明书，而是用场景和细节快速交代：

```json
{
  "start": {
    "chapterTitle": "第一章：大雾",
    "progress": 2,
    "ambient": "wind",
    "segments": [
      { "speaker": "系统", "text": "1997年，腊月。" },
      { "text": "大雾把整个县城盖住了，路灯像裹在纱布里。" },
      { "text": "你叫陈远，刚从省城回来，手里攥着一张没人等你的车票。" }
    ],
    "choices": [
      { "text": "往老街方向走", "next": "old_street_01" }
    ]
  }
}
```

建制的要素：
- **时间坐标**：哪个年代、什么季节/时间段（用系统旁白或一句环境描写带出，不要写"故事发生在1997年"这种说明文）
- **空间坐标**：在什么地方，这个地方长什么样、什么气味什么声音（用感官细节，不要抽象概括）
- **身份坐标**：玩家是谁，叫什么名字、什么身份、此刻在做什么（用一句叙述带出，不要写人物简历）
- **情境钩子**：玩家此刻面临的处境（为什么在这里、要干什么、什么事情正在发生或即将发生）

如果原作开头有这些信息，直接整理使用。如果原作开头是纯文学式的氛围起笔（比如一段景物描写），需要在保留氛围的同时，把建制信息嵌入前几个节点——可以分散在 2-3 个节点中，不必一个节点全部说完。

### 规则二：选择后的铺垫语

玩家做完一个选择后，不能直接跳进下一个叙事节拍。中间需要一段**铺垫语**——衔接上一个选择的后果和下一段内容的过渡。

**没有铺垫语的效果（节奏断裂）：**

```
节点A：你发现门没锁。
选择：[推门进去]
节点B：房间里有一个男人。
```

玩家会觉得：门开了、人就出现了？中间发生了什么？推门的时候看到了什么？

**有铺垫语的效果（节奏自然）：**

```
节点A：你发现门没锁。
选择：[推门进去]
节点B segments：
  "门很沉，推开的时候铰链响了一声。"
  "屋里的光线暗得几乎什么都看不见。你等了几秒钟，眼睛才适应过来。"
  "角落里坐着一个男人。"
```

**铺垫语的作用：**
1. 承接选择的动作本身（你推了门→门推开的过程和感受）
2. 交代空间/氛围的变化（从走廊进入房间→光线、气味、声音的变化）
3. 给玩家一个"缓冲带"，让选择的后果有重量感，而不是瞬间跳到下一个信息点

**执行规则：**

每个选择之后的目标节点，segments 的前 1-2 句应该是铺垫语，再进入正式内容。铺垫语可以是：
- 选择的动作结果（推门→门开了，什么声音、什么感觉）
- 环境/氛围的过渡（从一个空间到另一个空间的感官变化）
- 时间的过渡（如果选择导致了时间跳跃，用一句带过）
- 角色的即时反应（NPC 对玩家行为的第一反应——一个表情、一个动作，不是心理描写）

**但不要过度铺垫。** 铺垫语通常 1-2 句就够了。如果选择是轻量的（比如"继续听"），铺垫可以只有一句甚至省略。只有重大选择（路径分支、命运分支）的铺垫需要 2-3 句。

### 规则三：首次登场描写

关键角色、重要场景、关键道具在首次出现时，必须有一段简短的外貌/外观描写，让玩家在脑中建立视觉形象。

**为什么需要：** 文学作品中，角色外貌可能散落在全书各处，读者在阅读过程中逐渐拼出形象。但游戏是即时体验，玩家在看到一个角色名字的时候就需要"看见"这个人，否则这个角色在玩家心中永远是一团模糊的字母。

**首次登场描写的要求：**

1. **角色首次登场**：用 1-2 句 segments 描写外貌特征。不需要从头到脚，只需要抓住最有辨识度的 2-3 个视觉特征（体型、一个面部特征、一个穿着或姿态细节）。

```json
{ "text": "开门的是个瘦高个男人，头发乱得像刚睡醒，左手无名指上有一圈很深的戒痕——但没戴戒指。" }
```

2. **重要场景首次出现**：用 1-2 句描写空间的视觉特征和感官细节。跟 ambient 配合使用。

```json
{ "text": "这间办公室不大，窗帘拉着，日光灯把所有东西照得发白。桌上摞着三摞案卷，空气里有烟味和发霉的纸味。" }
```

3. **关键道具首次出现**：用半句到一句描写道具的外观。尤其是后续会影响剧情分支的道具，玩家必须"看见"它。

```json
{ "text": "抽屉最里面有一把钥匙，铜的，上面拴了一截红绳，绳子已经褪色了。" }
```

**描写来源：**
- 如果原作在其他位置有该角色/场景/道具的外貌描写，提取整理到首次出现的位置
- 如果原作完全没有外貌描写（有些作者就是不写外貌），根据角色的身份、时代、性格，结合剧情上下文合理想象
- 想象时遵守原作的风格锚点——如果原作是冷叙述，外貌描写也用冷叙述的方式，不要突然变成浓墨重彩的文学描写

**描写的禁忌：**
- 不要写清单式的外貌："他身高一米八，体重七十公斤，黑色短发，方脸"——这是档案不是叙事
- 不要写评价式的描写："她很美"、"他长得很凶"——让细节说话
- 不要一次描写太多：2-3 个特征就够了，其余的可以在后续节点中自然补充

**角色命名衔接（铁律）：**

角色的外貌描写和名字必须在玩家感知中形成明确的绑定。绝对不允许"上一个节点出现一个没名字的人，下一个节点突然冒出一个名字，玩家不知道是不是同一个人"。

三种处理方式：

**方式一：首次出场就带名字（最常用）**
```json
{ "text": "开门的是张远，瘦高个，头发乱得像刚睡醒。" }
```

**方式二：先外貌后命名（适合悬念/陌生感）**
外貌描写和命名必须在同一个节点内完成，或者命名时必须回扣外貌特征：
```json
// 节点A
{ "text": "柜台后面坐着一个男人，剃了光头，下巴上有一道疤。" }
{ "speaker": "光头男人", "text": "你找谁？" }
// 节点B（命名时回扣特征）
{ "text": "后来你才知道，那个下巴有疤的光头叫老邱。" }
```

**方式三：通过对白自我介绍**
```json
{ "text": "瘦高个男人伸出手。" },
{ "speaker": "瘦高个男人", "text": "张远。" }
// 之后的对白 speaker 切换为 "张远"
```

**禁止的写法：**
```json
// 节点A
{ "text": "一个男人站在门口。" }
// 节点B（玩家完全不知道张远是谁）
{ "speaker": "张远", "text": "你来了。" }
```
命名前用外貌特征作 speaker（如 `"speaker": "光头男人"`），命名后切换为真名。

### 规则四：场景切换使用 scene 字段

启动器支持 `scene` 字段，当 `scene.id` 变化时自动播放场景转场动画（场景名 + 描述 + 到达叙述）。**写作时必须利用这个机制来处理场景切换，而不是只靠 segments 里的文字。**

**scene 的三层内容各司其职：**

- `scene.name`：写地点，不写情绪。"厂医办公室"而不是"令人窒息的办公室"
- `scene.description`：写稳定不变的环境特征——光线、声音、空间关系、气味。这是玩家对这个空间的"第一印象底板"
- `scene.arrival`：写本次来到这个地方的动作或感受。同一个场景在不同节点被进入时，arrival 可以不同（第一次来 vs 第二次回来）

**场景切换的完整示例：**

```json
{
  "ch2_warehouse_01": {
    "scene": {
      "id": "warehouse",
      "name": "废弃仓库",
      "description": "铁门半开着，日光灯一闪一闪，水泥地上有柴油和铁锈的味道。",
      "arrival": "出租车在没有路灯的路上停下来。你付了钱，车尾灯消失在雨里。"
    },
    "ambient": "static",
    "segments": [
      { "text": "有人在里面说话，声音很低，听不清内容。" }
    ],
    "choices": [
      { "text": "走进去", "next": "warehouse_inside_01" },
      { "text": "先在外面听一会儿", "next": "warehouse_eavesdrop_01" }
    ]
  }
}
```

注意 arrival 承担了"怎么到的"这个过渡功能，description 承担了环境建立，segments 只需要写进入之后发生的事。三者分工，不重复。

**同一场景内连续推进时：**

保持相同的 `scene.id`，启动器不会重复触发转场。可以省略 description 和 arrival，或者只用简写形式：

```json
{ "scene": "废弃仓库" }
```

**大场景（major）切换频率控制：**

大场景切换会播放场景卡片动画，如果每隔三五个节点就触发一次，玩家会觉得不断被打断。**一个大场景至少应该持续 8-15 个节点再切换。**

判断标准：一个大场景就是一个"剧情发生的主要空间"——角色在这个空间里完成了一段有起承转合的事件。只有角色真正离开这个空间、去了另一个完全不同的地方时，才切换 major 场景。

同一空间内的小移动（从客厅走到阳台、从办公桌走到窗边、从仓库门口进入仓库内部），**不要切换 major 场景**。如果需要标记位置变化，用 `transient` 过场场景，或者保持同一个 `scene.id` 不变。

```
❌ 错误节奏（每 3-4 个节点就切 major）：
仓库门口(major) → 3个节点 → 仓库内部(major) → 2个节点 → 仓库二楼(major) → ...

✅ 正确节奏：
仓库(major, 持续12个节点) → 街道(major, 持续8个节点) → 警察局(major, 持续15个节点)
仓库内部的位置变化用 transient 或直接在 segments 中描写
```

**什么时候切换大场景（major）：**
- 角色从一个建筑/区域移动到另一个建筑/区域
- 章节转换时的场景变化
- 时间跳跃导致空间感完全不同（同一个房间但从白天变成深夜，如果空间感变化很大）

**什么时候用过场场景（transient）或不切换：**
- 同一建筑内的移动（楼上到楼下、房间到走廊）
- 短暂的闪回或回忆画面
- 快速推进中的中间地点（追逐、赶路中经过的地方）
- 一闪而过的瞬间（窗外的一幕、镜子里的倒影）

**什么时候不需要新 scene：**
- 同一空间内的小移动（从桌边到窗边）
- 对话过程中没有空间变化
- 紧张场景中的快速推进（追逐、搏斗中不需要停下来切场景）

### 规则五：支线质量底线——每条路径都必须看起来完整

**核心原则：玩家走任何一条路径，都不应该觉得自己被敷衍了。**

这条规则针对的是一种极常见的问题：主线写得很用心，但支线（尤其是草率结局线和短分支路径）明显偷工减料——两三句话就结束，没有铺垫没有过渡没有情绪收束，玩家一选就发现"完了？就这？"

**每条路径的最低完整度要求：**

**草率结局线（RASH ENDING）：**
- 不能只是"选错了→一句话→结局"
- 即使是"错误选择"导致的短结局，也必须有：铺垫（选择的后果开始显现）→ 发展（后果扩大）→ 收束（结局）
- 最少 3-5 个节点，让玩家经历一个虽然短但完整的因果链
- 结局描述不能只有一句话，至少写清楚：发生了什么后果、角色最终处于什么状态
- 草率结局的价值不是惩罚玩家，而是让玩家意识到"这条路走不通是有原因的"

**坏的草率结局（敷衍）：**
```
选择：不管那封信
结局：你错过了一切。（完）
```

**好的草率结局（短但完整）：**
```
选择：不管那封信 → 节点1（你把信塞回门缝，上楼倒了杯水）→ 
节点2（夜里风把信吹走了，你听到纸被风卷过走廊的声音但没起来看）→ 
节点3（第二天门缝里什么都没有，你站了一会儿才关门）→ 
结局：没人来过。也许从一开始，那封信就不是写给你的。
```

**路径分支：**
- 分出去的每条路径必须有自己的叙事弧线，哪怕很短
- 不能只是"绕了一圈又回到主线同一个点，中间什么也没发生"
- 如果路径最终汇合，汇合前的路径中必须发生了影响后续体验的事情（获得了线索、改变了关系、触发了flag、移动了val）
- 路径中的每个节点都应该有存在的叙事理由，不是凑节点数

**态度分支：**
- 虽然态度分支指向同一个 next，但选择不同态度后的下一个节点应该能感受到差异
- 至少通过铺垫语体现选择的不同后果："你直视他的眼睛" → 下一节点开头"他被你的目光逼退了半步" vs "你低下头不说话" → 下一节点开头"他等了一会儿，叹了口气"
- 如果两个态度选择之后的体验完全相同，这个选择就不应该存在

**分支质量自检：**

写完一条支线后，以玩家视角从头走一遍这条路径，问自己：
- [ ] 这条路径有没有自己的开头、发展、收束？
- [ ] 走完这条路径，玩家有没有获得新的信息、新的体验、或新的对角色的理解？
- [ ] 如果把这条路径删掉，游戏是不是完全不受影响？（如果是，说明这条路径没有存在的必要，要么加强它，要么删掉它）
- [ ] 玩家走完这条路径后会不会觉得"就这？"——如果会，说明内容量或质量不够

### 规则六：选择必须有回响（铁律）

**变量、flag、route、成就都不算选项 callback。callback 必须是玩家下一屏可读到的文本差异。**

这是整个技能中最容易被忽视的铁律。AI 很容易认为"我给选项加了 changes（val+10、addFlag），所以选择有后果了"。错。玩家看不见变量面板的变化，看不见 flag 的设置。**如果选了 A 和选了 B 之后，玩家读到的下一屏文字完全一样，那这个选择就是假的。**

**同向选项的 callback 规则（最高频违规）：**

当多个选项指向同一个 next 节点时，必须在中间插入独立的 callback 节点——每个选项各一个，包含针对该选择的 1-2 句文本反馈，然后再跳到共同的 next。

```
❌ 错误（两个选项直接指向同一节点）：
选项A → node_05
选项B → node_05
（node_05 的内容跟选了A还是B完全无关）

✅ 正确（每个选项有独立 callback）：
选项A → callback_A（1-2句针对A的反应）→ node_05
选项B → callback_B（1-2句针对B的反应）→ node_05
```

callback 节点的要求：
- 至少 1 个 segment，内容必须回应选项的具体动作/台词
- callback 的 next 指向原来共享的目标节点
- callback 的文本必须让玩家感觉到"故事注意到了我的选择"

**结局前收束规则：**

选项不得直接跳到结局节点。结局前必须有至少一个收束节点，包含：人物反应、关系状态变化、行动后果落地、或意象回收。

```
❌ 选项 → ending_true（直接跳结局）
✅ 选项 → closure_node（收束：后果落地）→ ending_true
```

**问题的三种表现：**

1. **选择之后没有反应**——玩家选了"质问他"，下一个节点的内容跟选"沉默不语"完全一样。
2. **做了和原作不同的选择，后续对白/叙述没变**——玩家选了原作中角色没有走的那条路，但接下来读到的段落仍然是原作的文本，逻辑完全对不上。
3. **选择的影响只存在一个节点就消失**——玩家选了激怒对方，下一个节点对方确实生气了，但再下一个节点对方又恢复正常了，好像什么都没发生。

**三层回响机制：**

**第一层：即时反应（必做）**

**三层回响：**

**即时 callback（每个选择必做）**——已在上方详细说明。核心：每个选择的下一屏必须有针对该选择的文本差异。changes 不算 callback。

**持续影响（关键选择必做）**——重要选择的影响通过 flag/变量在后续多个节点中产生可感知差异：NPC 态度变化、可见选项增减、叙述细节变化。

**结局差异（整体必做）**——玩家的选择组合最终导向不同结局，通过 flag 路由实现。

**原作文本的改写铁律：**

当玩家做了和原作主角不同的选择时，**后续内容必须真正偏离原作文本**。这意味着：

- 不能继续使用原作的对白和叙述——逻辑已经不通了
- 必须基于"如果角色做了不同选择，这个世界会怎么反应"来写新内容
- 新内容仍然遵守风格锚点和角色卡（风格不变，但事件变了）
- 如果分支最终要汇合回主线，汇合的方式必须合理——不能"不管你选了什么，三步之后一切回归原样"。要通过一个新的事件或情境把岔开的路合理地拉回来，并且玩家携带的状态已经不同了

**选择回响自检：**

每写完一个包含选择的节点，立刻检查：
- [ ] 每个选项指向的下一个节点，内容是否针对该选择有专属反应？
- [ ] 如果两个选项指向同一个节点，节点内部是否有区分不同来源的内容？如果没有区分机制，是否应该改为不同节点？
- [ ] 关键选择是否设置了 flag 或变量？后续是否有节点引用了这些 flag/变量来展示差异？
- [ ] 如果这个选择偏离了原作路径，后续 2-3 个节点的内容是否真正脱离了原作文本？

### 规则七：分支信息隔离（铁律）

**每条分支路径必须在信息上自给自足。玩家在路径A中获知的信息，绝对不能在路径B中被默认为已知。**

这是测试中反复出现的严重问题：某个角色的名字、某个地点的用途、某个事件的真相，只在选项A的路径中被建立，但选了选项B的玩家在后续节点中会突然遭遇这个信息被当作"大家都知道"来使用。

**信息隔离的具体要求：**

**1. 专有名词的独立建立**

任何专有名词（人名、地名、组织名、术语）在每一条可能到达它的路径中都必须独立建立。不能因为"另一条路径里已经解释过了"就省略。

```
❌ 错误：
选项A路径 → 节点X（介绍"老邱是厂里的保安"）
选项B路径 → 节点Y（直接写"老邱说……"，但B路径的玩家从未见过老邱）

✅ 正确：
选项A路径 → 节点X（介绍老邱）
选项B路径 → 节点Y（老邱在B路径中也有自己的登场和介绍）
如果B路径中老邱不出现，那B路径中就不应该提到老邱
```

**2. 事件信息的路径归属**

如果某个事件只在路径A中发生，那么路径B中的所有后续节点都不能引用这个事件的结果。即使两条路径在后面汇合了，汇合后的节点如果要引用路径A的事件，必须用 condition（`hasFlag`）检查玩家是否走过路径A。

**3. 汇合节点的信息处理**

当多条路径汇合到同一个节点时，这个节点的内容只能使用所有路径都共有的信息。路径特有的信息必须通过 condition/flag 差异化呈现。

如果汇合节点无法做到差异化（比如引擎不支持 segment 级 condition），那就不要让需要引用路径特有信息的内容出现在汇合节点——要么改为不同节点，要么把路径特有信息的引用放到有 condition 控制的选项中。

### 强制自检协议（铁律）

**结构合格不等于互动叙事合格。自检必须以玩家视角为出发点，但不需要逐屏完整走查——聚焦高风险节点。**

**每写完一个章节，执行以下两步。不通过不进入下一章。**

#### 步骤一：高风险节点定向检查

只查最容易出问题的四类节点，不需要逐个节点走查：

**A. 同向选项节点**（多个选项指向同一 next）
→ 是否有独立 callback 节点？callback 文本是否回应了具体选择？

**B. 结局前一步的节点**（next 或 routes 指向 ending 的节点）
→ 是否有收束节点？不能直接跳结局。

**C. 每个选项**（逐条扫描本章所有选项文本）
→ 格式：台词/动作？有没有"说XX"？
→ 名词：每个专有名词在前文建立过？指不出来→修
→ 情境：和 segments 最后一句对得上？

**D. 场景切换节点**（scene.id 发生变化的节点）
→ 第一句是否衔接了上一个节点？有没有跳跃？

**E. 叙事目的覆盖**
→ 对照事件链，本章所有原作事件的叙事目的是否都在游戏节点中保留？
→ 新增的分支节点是否都有明确的叙事目的？
→ 有没有"空转"节点（找不到叙事目的的节点）？

#### 步骤二：内容质量十问

对本章整体回答以下十个问题，任何一个答"否"都必须修复：

1. 每个选项是否只引用当前节点已经出现的信息？
2. 每个选项是否是当场可做的具体动作/台词，而不是抽象态度？
3. 点下每个选项后，下一屏是否有文本 callback，而不是只改变量？
4. 草率结局是否写出了世界变化（当场反应→社会后果→人物态度→物件归处→判词），而不只是"提前退场"？
5. 正式结局是否有长线后果收束，而不是 ending 页交代剧情？
6. 结局页是否是判词，而不是剧情摘要？
7. 有没有任何文字跳出故事世界（"玩家"、"节点"、"分支"、模板句式）？
8. 有没有为了体量而重复同类选择、同类句式、同类场景？
9. 每条非原著延伸是否像原著世界可能长出来的，而不是作者突然发明？
10. 玩家点到任何一个位置，会不会觉得自己的选择被认真对待？

```markdown
### 第N章 自检报告
- 同向选项：X处，callback 完整/缺失X处（已修复）
- 结局收束：X处，收束节点 完整/缺失X处
- 选项扫描：格式问题X处，名词违规X处（已修复）
- 内容完整性：事件覆盖X/X，金句X/X
```

## 互动节奏：生死线

### 硬性规则

**每 1-5 句正文之后，必须给玩家一次互动机会。**

互动的形式可以是：
- 真分支（2-4 个选项，通向不同路径或改变不同数值）
- 单选确认（只有一个选项，如"继续"、"走过去"、"打开它"）
- 条件选项（根据 flag/val 显示不同选项）

### 什么时候用单选确认

单选确认不是偷懒——它维持玩家的参与感。适合用在：
- 叙述段之间的自然停顿
- 场景转换之间
- 一段氛围铺设之后，让玩家"确认进入"
- 紧张场景中的动作推进（"推开门"、"接起电话"）

### 什么时候允许超过 5 句

只有以下场景允许长段落（6-10句）：
- 结局的最终叙述
- 独白高潮
- 真相揭露的关键段落
- 审判/告白/决战等高密度场景

即使在这些场景中，也建议在第 7-8 句左右给一个确认互动。

## 节点写作清单

写每个节点时，按以下清单逐项完成：

### 0. 叙事目的确认（每个节点的第一步）

**在写任何一个节点之前，先回答：这个节点为什么必须存在？删掉它游戏会缺少什么？**

对照事件链中该事件的叙事目的，确认这个节点要完成什么：交代信息？推进情节？建立角色？埋伏笔？回扣伏笔？呈现主题？提供呼吸？

- 原作事件对应的节点：叙事目的从事件链继承，不能丢失
- 分支路径新增的节点：必须有独立的叙事目的（推进分支线的因果链、建立分支独有的角色互动等）
- 过渡节点：叙事目的是"节奏衔接"——虽然功能是连接，但不能空转
- callback 节点：叙事目的是"选择回响"——让玩家感知到世界对选择的反应

**如果一个节点找不到叙事目的 → 它不应该存在。合并到相邻节点或删除。**

### 1. 节点 ID
- 使用之前分支拓扑文档中预设的 ID
- 格式：`章节_场景_序号`，如 `ch1_letter_01`、`ch2_rooftop_03`
- 结局节点：`ending_` 前缀

### 2. chapterTitle（章节开始节点）
- 如果某个节点是章节/幕的开头，必须写 `chapterTitle`
- 用于触发章节过渡动画，给玩家"进入新章节"的仪式感
- 只在章节第一个节点设置，不要每个节点都设
- chapterTitle 的文字应该有氛围感，不要只写"第二章"，而是"第二章：潮声之前"

示例：
```json
{
  "chapter_02_start": {
    "chapterTitle": "第二章：潮声之前",
    "segments": [
      { "text": "第二天清晨，雨停了。" }
    ],
    "choices": [
      { "text": "走向车站", "next": "station_001" }
    ]
  }
}
```

注意：当分支导致玩家可能从不同路径进入同一章时，确保所有入口路径都指向带 chapterTitle 的同一个节点，避免玩家跳过章节过渡。

### 3. scene 场景
- 每个有正文的节点都建议写 scene（保持相同 id 不会重复触发转场）
- `type: "major"`（默认）：大场景，切换时显示场景卡片。**一个 major 场景至少持续 8-15 个节点再切换**，不要频繁打断
- `type: "transient"`：过场场景，只更新位置不播动画。同一建筑内移动、闪回、快速推进用这个
- 同一空间内的小移动（客厅→阳台、门口→内部）不要切 major，用 transient 或保持同一 id
- `name` 写地点不写情绪
- `description` 写稳定环境特征，不写剧情事件
- `arrival` 写本次进入场景的动作/叙述

### 4. progress
- 估算当前节点在全流程中的进度百分比
- 第一个节点通常是 5
- 每个结局节点是 100
- 中间节点按比例分配
- 在同一章的路径分支中，平行路径的 progress 应该大致相同

### 5. segments 正文

**写作规则：**

- 每句叙述尽量简短，服务于推进节奏
- 对白必须带 speaker 字段
- 系统旁白用 `"speaker": "系统"` 或 `"speaker": "旁白"`
- 不写心理描写：禁止"他意识到"、"她领会到"、"其实是XXX"
- 不写说教和强行催泪
- 对白口语化，不要过度比喻和书面语
- 不要在 text 中使用 HTML 标签

**段落数量控制：**
- 普通推进节点：2-4 个 segments
- 氛围铺设节点：3-5 个 segments
- 关键剧情节点：4-6 个 segments（这已经是上限，到了必须给互动）
- 结局节点：不用 segments，用 description 字段

**speaker 的使用：**
- 对白必须标注 speaker，使用角色名
- 叙述段不写 speaker（留空或不设该字段）
- 环境描述、旁白用 `"speaker": "系统"` 或 `"speaker": "旁白"`
- 信件、纸条等用物品名作 speaker：`"speaker": "信"`, `"speaker": "纸条"`

**原作体量保持（铁律）：**

**原作中的每一段情节，在游戏的对应路径中必须保持至少同样的体量。只允许横向扩展（添加分支），不允许纵向缩略（压缩原文内容）。**

这条规则针对AI改写中最常见的问题：原作用了一整页写一场对话，AI改成游戏后只剩三句话概括。原作花了两个段落描写一个场景，AI用一句"你来到了一个旧仓库"带过。

具体要求：
- 原作中一段 5 轮对话，游戏中这段对话至少也要 5 轮（可以更多但不能更少）
- 原作中一个场景的环境描写有 3 句，游戏中 scene.description + segments 加起来不能少于 3 句的信息量
- 原作中一个事件的发展过程经过了 4 个步骤，游戏中这个事件至少也要 4 个步骤（可以拆成更多节点但不能合并跳过）
- 如果原作在某处写得很"满"（对白密集、描写细致、节奏放慢），游戏中这个位置也必须"满"，不能因为"要控制节点数量"而压缩

**唯一允许缩略的情况**：原作中纯心理描写的段落（"他想到了……他意识到……"），因为游戏禁止心理描写，这部分需要转化为外在可呈现的形式（动作、对白、环境），转化后的体量可以适当调整。

**金句与关键句保留（铁律）：**

第一步索引中摘录的金句与关键句，在写作时必须原样保留在对应节点的 segments 中。不改写、不用自己的话替代、不缩略。

- 金句出现在 segments 中时，保持原文用词
- 如果金句是对白，保持原文的 speaker 和完整句子
- 如果金句所在场景被分支覆盖，金句应出现在主线/真结局路径中
- 金句可以作为某个节点的"情绪锚点"——在这句话前后适当留白（用单选确认给一个呼吸），让玩家能感受到这句话的分量

### 6. choices 选项

**选项数量：**
- 多数情况用 **2 个选项**——两个方向足以构成有意义的选择张力
- 少数关键节点可以 **3 个选项**——当有三种确实不同的方向时
- **不要出现 4 个选项**——选择太多玩家会选择困难，且每个选项的后果都需要独立开发，4 个会导致内容量失控
- 单选确认 1 个（"继续"、"走过去"）

**选项设计铁律：**

**1. 选项中不能出现玩家尚未获知的信息**

选项文本里的每一个名词、每一个概念，玩家在当前节点之前必须已经知道。如果玩家不知道"老邱"是谁，选项就不能写"去找老邱"。如果玩家不知道有一个"后门"，选项就不能写"从后门离开"。

```
❌ "去找老邱问清楚"（玩家还不知道老邱是谁）
❌ "打开保险箱"（前文从未提到过保险箱的存在）
❌ "使用应急通道"（玩家不知道有应急通道）

✅ "去找刚才那个光头"（玩家见过但还不知道名字）
✅ "试试桌上那把钥匙"（前文已经描写过钥匙）
✅ "从你进来的那扇门离开"（玩家经历过进门）
```

**2. 每个选项都必须看起来合理**

不能有一个选项明显是"正确答案"而另一个明显是"送死"。好的选择是两个方向都有道理，玩家基于自己的判断和倾向做决定，而不是基于"哪个不会死"做决定。

```
❌ "冷静地分析局势" vs "发疯大叫"（一个明显理性一个明显荒谬）
❌ "救他" vs "看他去死"（道德绑架，没有真实的选择张力）

✅ "留下来陪他" vs "先出去找人帮忙"（两个都合理，取舍不同）
✅ "直接问她" vs "先不说，观察几天"（两种策略，各有道理）
```

**3. 选项文本格式（绝对铁律，零容忍）**

**说话类选项必须直接写台词原文，用双引号。动作类选项直接写动作。**

**绝对禁止以下所有写法：**
- ❌ "也说早安" → 这是描述行为，不是台词
- ❌ "说早安，然后声明不会接受分配" → 描述行为 + 塞入后续剧情
- ❌ "让开门口，请她进来" → 描述行为
- ❌ "说今天批了假，周小姐要来" → 描述行为 + 引用未知角色
- ❌ "问他是不是急着搬走" → 描述行为
- ❌ "问他为什么来" → 描述行为
- ❌ "说出你的想法" → 描述行为
- ❌ "告诉她真相" → 描述行为
- ❌ "直接声明不会接受" → 描述行为
- ❌ "表示同意" / "表达愤怒" / "选择沉默" → 抽象描述
- ❌ "回应她的问题" → 描述行为
- ❌ "也打了个招呼" → 描述行为

**必须写成的样子：**
- ✅ "早上好。"（直接台词）
- ✅ "你怎么来了？"（直接台词）
- ✅ "我知道那件事。"（直接台词）
- ✅ "我不同意。"（直接台词）
- ✅ "今天不上班。"（直接台词，不提未出场的角色）
- ✅ "你要搬走？"（直接台词）
- ✅ "……"（沉默也是一种台词）
- ✅ 转身离开（动作类，不加引号）
- ✅ 侧身让她进来（动作类）
- ✅ 盯着他手肘下夹的东西（动作类，具体可见的行为）

**判断标准**：选项文本能不能直接被演员说出来或做出来？"也说早安"演员没法演——他不知道该说什么。"早上好。"演员可以直接说。"直接声明不会接受计划爱情"演员没法演——这是导演的指令不是台词。"我不会接受安排好的感情。"演员可以直接说。

**4. 每个选项必须通向有意义的后果**

不允许存在"装饰性选项"——看起来是选择，但选了之后和另一个选项的体验完全一样。如果一个选项无法带来任何差异（不改变叙述、不改变数值、不改变后续选项），就不应该存在。

**5. 选项必须回应当前情境（铁律）**

选项是玩家对"此刻正在发生的事"的直接反应。每个选项都必须通过以下三重检查：

**检查一：情境关联**——选项是否直接回应了 segments 最后呈现的情况？

```
❌ segments：周小姐说"早安。" → 选项："直接声明不会接受计划爱情"
   （对方只是说了句早安，这个选项完全脱离当前情境，把后续剧情的内容提前塞进了选项）
❌ segments："你站在雨里，门锁着。" → 选项："去找张远"（跳到完全无关的事）

✅ segments：周小姐说"早安。" → 选项："早上好。" / 低头不说话
✅ segments："你站在雨里，门锁着。" → 选项："踹门" / "绕到后面看看"
```

**特别注意：不要把后续剧情的信息或冲突提前塞进当前选项。** 如果故事后面会涉及"计划爱情"的冲突，那个冲突应该在后续节点中通过对白和事件自然展开，而不是在角色刚说"早安"的时候就作为选项跳出来。选项只回应此刻，不预知未来。

**检查二：信息基础**——选项中提到的每个人、物、地点、概念，玩家是否都已经知道？

```
❌ "说今天批了假，周小姐要来"（周小姐从未在游戏中出现过，玩家不知道她是谁）
❌ "说早安，然后声明不会接受分配"（"分配"这个概念还没在游戏中建立）
❌ "用后门的密码锁"（玩家不知道后门有密码锁）

✅ "今天不上班，请了假。"（只说玩家已知的事实）
✅ "试试刚才在抽屉里看到的那串数字"（前文已建立）
```

**小说叙事手法不适用于选项（铁律）：**

原作小说可能使用倒叙、悬念前置、信息不对称等手法——读者在第一章就看到"周小姐"这个名字，但要到第三章才知道她是谁。这在阅读中制造悬念。**但游戏选项必须严格基于主角当前的已知信息。** 主角此刻不知道的人名、不了解的概念、还没发生的事件，都不能出现在选项中。

原作的倒叙和前置信息应该在游戏的 segments（叙述段）中通过 scene、对白、旁白等方式逐步建立，而不是直接扔进选项让玩家选一个自己不理解的东西。

**检查三：因果合理**——从 segments 的情境出发，玩家做出这个选择的心理动机是否成立？

```
❌ segments："他突然对你笑了。" → 选项："打他"（没有任何攻击动机）
✅ segments："他突然对你笑了。" → 选项："笑着回应" / "你笑什么？"
```

**自检方法**：遮住 segments 只看选项——能猜出玩家面对什么情况吗？遮住选项只看 segments——选项是这个情境下的自然反应吗？两个方向都要通过。

**6. 选项必须在主角的决定权范围内**

玩家扮演的是一个具体的角色，这个角色有自己的身份、能力和权限边界。选项中的动作必须是这个角色此刻有能力、有权限、有条件执行的。

```
❌ 主角是普通工人 → 选项："命令警察逮捕他"（没有这个权力）
❌ 主角在三楼 → 选项："从窗户跳下去追他"（不合常理的能力）
❌ 主角不认识对方 → 选项："说出他的秘密"（不掌握这个信息）
❌ 主角被绑着 → 选项："拿起桌上的刀"（物理上做不到）

✅ 主角是普通工人 → 选项："偷偷跟上去" / "去告诉厂长"
✅ 主角在三楼 → 选项："跑下楼追" / "在窗口记住他的样子"
✅ 主角不认识对方 → 选项："你是谁？" / "假装没看见"
✅ 主角被绑着 → 选项："挣扎试试绳子松不松" / "大声喊人"
```

每个选项写完后问自己：**这个角色在这个处境下，真的能做这件事吗？** 能力上做得到、信息上知道、物理条件允许、身份权限覆盖——四项全过才合格。

**7. 选项只能引用当前节点可见的内容**

选项中的人名、物件、地点，必须在**当前节点的 segments 或 scene** 中出现过，不能仅仅因为"同一场景前面的节点提到过"就在当前节点的选项里用。玩家的注意力在当前屏幕上，不会翻回前面确认。

```
❌ 当前节点正文只提到了陈一单和走廊 → 选项提到"酒杯"（酒杯在前面的节点出现过但当前节点没有）
✅ 当前节点正文提到了桌上有一个杯子 → 选项："拿起杯子"
```

**8. 禁止模板痕迹**

以下词汇/句式绝对不能出现在选项和正文中（验证脚本会自动检测报错）：

选项中禁止：
- "把事情推到极端"、"把话咽回去"
- "顺着XX继续问"、"带着XX往后走"
- "关于XX的那句话"、"XX的余劲"

正文中禁止：
- "玩家"、"节点"、"分支"、"支线"、"选项A/B"
- 任何暴露"这是一个游戏"的元叙事词汇

这些是生成器露馅的标志。一旦出现，玩家会立刻出戏。

**condition 的使用：**
- 根据 flag/变量/val 控制选项的显示
- 确保在所有可能的状态下，至少有一个选项可见
- 建议最后一个选项用 `"condition": "default"` 兜底

## 节点逻辑连贯性（铁律）

**每一个节点和它的前后节点之间必须存在可感知的逻辑关联。每一个选项和它所在的情境之间必须存在直接的因果关系。** 不允许任何"读了之后不知道为什么到了这里"或"不知道这个选项和刚才的事有什么关系"的断裂。

### 前向连贯：本节点第一句必须衔接上文

本节点 segments 的第一句，和上一个节点的最后一句（或选项文本）之间，必须有以下至少一种逻辑关系：

- **动作延续**：上文"你推开门" → 本节点"门后面是一间空荡荡的办公室"
- **对白延续**：上文角色A说了话 → 本节点角色B回应或主角回应
- **时间延续**：上文"天开始黑了" → 本节点"路灯亮了起来"
- **因果延续**：上文做了选择 → 本节点展示选择的即时后果
- **空间延续**：上文在地点A → 本节点通过 scene.arrival 交代了移动过程 → 到达地点B

**常见断裂示例（全部禁止）：**

```
❌ 上一节点："他把信递给你。" → 本节点："你在街上走着。"
   （信怎么处理了？怎么到街上的？）

❌ 选项："质问他为什么骗你" → 下一节点："第二天早上你醒来。"
   （质问的过程呢？他怎么回应的？）

❌ 上一节点：角色A正在说一件重要的事 → 本节点：突然切到另一件完全无关的事
   （A说完了吗？说的内容有什么影响？）

❌ 选项："跟着她走" → 下一节点："你坐在办公室里。"
   （跟着走到了哪里？怎么到了办公室？）
```

### 后向连贯：选项→下一节点的因果链

选项的 next 指向的节点，其开头内容必须是该选项动作的直接后续。中间不能跳步。

**逐选项检查流程：**

每写完一个包含选项的节点，对每个选项执行以下检查：

```
选项文本："踹门"
  → 下一节点第一句是否描写了踹门的结果？（门开了/门没开/脚疼了）
  → 如果下一节点跳到了其他内容，中间发生了什么？补上。

选项文本："你凭什么？"
  → 下一节点是否是对方听到这句话后的反应？（表情变化/回话/沉默）
  → 如果下一节点完全没有对方的反应，这个选项就是断裂的。

选项文本："转身离开"
  → 下一节点是否描写了离开的过程或离开后到达的地方？
  → 如果下一节点直接跳到一个完全不同的场景且没有移动过渡，补上。
```

### 跨选项连贯检查

写完一个有多个选项的节点后，把所有选项及其后续节点放在一起对比：

- [ ] 每个选项指向的下一节点，第一句是否直接承接了该选项的动作/对白？
- [ ] 如果多个选项指向同一个节点，该节点是否通过变量/flag 差异化处理了不同来源？
- [ ] 下一节点中是否引用了只在其他选项路径中才能获知的信息？
- [ ] 每个选项到下一节点之间是否存在未交代的时间/空间跳跃？

### 7. changes 状态变动

每个选项的 changes 应该包含：

```json
{
  "changes": {
    "val": 10,                           // val 增减（可选）
    "set": { "variable": "value" },       // 变量赋值（可选）
    "addFlag": "flag_name",               // 添加 flag（可选）
    "unlockAchievement": "achievement_id" // 解锁成就（可选）
  }
}
```

原则：
- 不是每个选项都需要改变所有类型的状态
- 态度分支：通常只改 val
- 路径分支：改 val + 设 flag（标记走了这条路）
- 命运分支：改 val + 设 flag + 可能设变量
- 成就解锁放在有意义的时刻，不要随手加

### 8. ambient / theme

- ambient 是章节级的氛围设置，在章节的第一个节点或氛围发生明显变化时设置
- theme 很少在节点级覆盖，除非剧情需要（如闪回、梦境、恐惧场景）
- 不要每个节点都换 ambient

### 9. effect / effects

- 特效只用在关键瞬间
- 适合用特效的时刻：震惊发现、危险降临、时间扭曲、记忆闪回、世界崩塌、环境突变
- 一个节点通常最多 1 个 effect，重大场景最多 2 个
- 选项也可以带 effect（点击时触发），适合用在"做了就没法收回"的选择上

## 自动分批写作协议

**核心原则：宁可分多次对话完整输出，也不压缩内容塞进一次。**

### 分批判定与输出规则

**每次最多写 50-100 个节点。** 每批输出后暂停，告知进度，等待用户"继续"。

写作过程中必须反复参阅章节设计文档中的节点规划和叙事目的清单。每写完 5-8 个节点回看一次，确认不偏离。

| 条件 | 处理 |
|------|------|
| 预估 ≤100 节点且原作 <3000 字 | 一次完成 |
| 其他所有情况 | **必须分批**，每批 50-100 节点 |

### 跨对话长记忆管理（铁律）

分批写作必然跨越多次对话。**每次新对话开始时，AI 的工作记忆是空的——之前对话中的所有规划、角色卡、风格锚点、状态追踪全部丢失。** 如果不做强制恢复，后续章节的质量必然断崖式下降（遗忘角色说话方式、丢失 flag 追踪、风格漂移、情节跳过）。

#### 每次对话结束时必须输出的「断点存档」

```markdown
# 断点存档：《作品名》 至第N章

## 一、全局文档清单（下次对话必须带入）
1. 原作索引文档（骨架层+人物层+场景层+衔接标记）
2. 风格锚点文档
3. 分支文学大纲
4. 状态与结局设计文档
5. 已完成的 JSON 文件

## 二、当前进度
- 已完成章节：第1-N章（共XX个节点）
- 下一章：第N+1章
- 全书总进度：XX%

## 三、累积状态快照
- val 当前可能范围：XX-XX
- 已设置的所有 flag：[完整列表]
- 已设置的变量：{完整列表}
- 已解锁的成就：[列表]
- 已触发的草率结局：X个（目标X个）

## 四、下一章预告
- 下一章对应原作事件：E_XX 至 E_XX
- 下一章关键角色：[谁会出场]
- 下一章预计分支点：[简述]
- 下一章需要注意的衔接：上一章末尾的状态如何接入

## 五、质量基线记录
- 前N章平均每章节点数：XX
- 前N章对白密度参考：每个节点平均X轮对白
- 前N章分支路径平均长度：X个节点
（后续章节不得低于这些基线）
```

#### 每次新对话恢复时必须执行的「记忆重建」

1. **加载全局文档**：索引、风格锚点、文学大纲、状态设计（用户提供或从上次存档复制）
2. **核对累积状态**：逐项核对断点存档中的 flag、变量、val 范围，确认与已完成 JSON 一致
3. **重读上一章最后 3-5 个节点**：恢复叙事连贯性的手感，确保衔接
4. **重读角色卡中本章涉及角色的说话方式**：防止风格漂移
5. **核对质量基线**：确认本次对话的产出不低于前面章节的密度

**记忆重建不是可选的。** 即使用户直接说"继续写"，也必须先完成上述五步。可以不向用户展示重建过程，但必须实际执行。

#### 跨对话质量防退化（铁律）

**后续章节的质量不得低于前面章节。** 具体指标：

- 每章节点数不得低于前N章平均值的 80%
- 对白密度不得低于前N章水平
- 分支路径长度不得低于前N章平均值
- 选项质量标准（格式、情境、callback）完全一致
- 自检协议完全一致——不能因为"后面的章节赶时间"就简化自检

**常见的退化模式（全部禁止）：**
- ❌ 前三章每章 80 个节点，第七章突然变成 30 个
- ❌ 前三章每个选择都有 callback，第七章开始共享节点无差异
- ❌ 前三章对白写 5-6 轮，第七章对白只写 2 轮
- ❌ 前三章有过渡衔接，第七章开始跳切
- ❌ 前三章做完整自检，第七章跳过自检直接输出

### 反压缩铁律（无例外）

**任何包含信息的内容绝不允许压缩。最常见的压缩不是"缩短"，而是"直接不写"——整段情节跳过、关键信息省略、过场衔接消失、对白轮次砍掉。**

唯一允许精简：百万字级网文中无信息的水词。不精简任何承载角色性格、情节推进或氛围建立的段落。

**禁止清单（按发生频率排序）：**
- ❌ **整段情节直接跳过不写**（最常见）
- ❌ **过场和衔接直接省略**（第二常见）
- ❌ **对白轮次砍掉只留一两句**（第三常见）
- ❌ 缩减节点数
- ❌ 跳过分支路径只写主线
- ❌ 省略场景描写
- ❌ 草率结局只有 1-2 个节点
- ❌ 缩短分支结局线
- ❌ 用"他们聊了一会儿"替代完整对话
- ❌ 用"几天后"跳过有具体描写的时间段

**压缩检测**：如果正在写的章节比文学大纲中规划的节点数少了 30% 以上 → 正在压缩 → 立刻停下，按章节边界分批交付。

### 章节完整性

一个章节（含所有分支和草率结局线）必须一次输出完整，不从中间断开。

### 每章的嵌套工作流（铁律）

**每个章节在写 JSON 之前，必须先生成一份「章节设计文档」。这份文档是次级记忆——它在本章写作全程常驻上下文，确保本章内部的结构、分支、状态、节奏全部有据可循，不会在写作过程中遗忘或偏离。**

#### 触发条件

**章节设计文档是 AI 的内部工作文档，必须实际生成，不能因为用户没有要求就跳过。** 即使用户说"直接写 JSON"，也必须先在内部完成本章的设计文档再动笔。可以不向用户展示全文，但必须实际生成并在写作中引用。

| 情况 | 文档版本 | 说明 |
|------|---------------------|------|
| 短篇（<1万字）且本章预估 ≤15 个节点 | **简版**：只写第三、四项（改编结构 + 状态系统） | 短篇章节简单，不需要完整七项 |
| 短篇但本章有 2 个以上分支点 | **完整版** | 分支多则需要完整记忆防止混乱 |
| 中篇（1-5万字）任何章节 | **完整版** | 中篇必须每章完整规划 |
| 长篇（>5万字）任何章节 | **完整版** | 长篇是记忆衰减的重灾区 |
| 任何体量，但本章预估 >25 个节点 | **完整版 + 分段核对**：每 5-8 个节点强制回看 | 节点多时记忆衰减最严重 |
| 断点恢复后的第一章 | **完整版 + 上下文重建**：额外复述前文关键状态 | 跨对话恢复时上下文全部丢失 |

**简版章节设计文档**只需要：
```markdown
# 第N章 简版设计

## 改编结构
- 主线路径：...
- 分支点：...

## 状态系统
- val 变化：...
- 新增 flag：...
```

**完整版**包含全部七项（见下方模板）。

#### 章节设计文档（次级记忆，铁律级必须生成）

```markdown
# 章节设计文档：第N章 「章节标题」

## 一、本章在全书中的位置
- 全书进度：XX% - XX%
- 节奏区域：○慢 / ●快 / ◐停顿 / ◑转折（从全书节奏曲线中定位）
- 前一章交接状态：（从上一章交接清单复制）

## 二、本章原作内容摘要
- 原作事件链（本章对应的 E 编号范围）：
  E12（交代+建立）：XXX
  E13（推进）：XXX
  E14（伏笔）：XXX
  ...
- 本章叙事目的清单（从事件链继承）：
  · 交代类：X个（哪些信息必须在本章向玩家建立）
  · 推进类：X个（哪些情节必须推进）
  · 伏笔类：X个（哪些伏笔要埋/回扣）
  · 主题类：X个（哪些主题呈现不能丢）
- 本章涉及的原作金句：（编号 + 原文 + 预计放置节点）

## 三、本章改编结构
- 主线路径：（事件 → 事件 → 事件，线性描述本章主线经过哪些节拍）
- 分支点设计：
  · 分支点1（节点ID）：选项A → 去向 / 选项B → 去向 / 分支类型
  · 分支点2（节点ID）：...
- 路径分支展开：
  · 分支路径A：（2-3句描述这条路上发生什么，最终汇合到哪里或通向什么结局）
  · 分支路径B：...
- 草率结局入口（如在前60%）：
  · 触发选择 → 短线发展 → 草率结局方向

## 四、本章状态系统
- val 变化规划：
  · 节点XX：+10（原因）
  · 节点YY：-15（原因）
  · 本章 val 波动范围：从约XX到约XX
- 新增 flag：
  · flag_name：在哪个节点设置 / 普通flag 还是 importantFlag
- 新增变量变动：
  · variable_name：赋什么值 / 在哪个节点

## 五、本章场景规划

- transient 场景：（哪些过渡位置用）
```

#### 为什么需要这份文档

长内容写作中，AI 的工作记忆会在写了十几个节点后开始模糊。没有这份文档，会出现：
- 写到章节中段忘记前面设了什么 flag，后面引用不上
- 分支路径写到一半忘记要汇合到哪里
- val 的变化方向和前面规划的不一致
- 金句被遗忘，该出现的位置跳过了
- 过渡节点被遗忘，衔接标记中"需补"的没补

选项中引用未知信息的问题通过自检二的「逆向追溯检查」解决——不需要预建表格，在每章自检时对每个选项做反查即可。

这份文档就是防止这些问题的"备忘锚"。写每个节点时，回看这份文档确认当前进度。

#### 执行流程

```
1. 加载上下文：全局文档（索引+风格+大纲+状态设计）+ 上一章交接清单
2. 生成本章「章节设计文档」（次级记忆）
3. 带着章节设计文档写作本章所有节点的 JSON
4. 每写完 5-8 个节点，回看章节设计文档核对进度
5. 全部节点写完后执行四项强制自检
6. 发现问题立即修正
7. 生成交接清单
8. 将本章节点追加到总 JSON
9. 进入下一章（不暂停）
```

### 章节交接清单

```markdown
## 第N章 交接清单

### 节点 ID 列表 + 入口/出口节点
### 本章结束时的状态范围（val / flags / variables）
### 本章新增成就
### 草率结局进度（前60%内是否已设置？已设几个？）
```

### 上下文保护（铁律）

上下文即将耗尽时主动停下，保存已完成 JSON + 恢复文档。**绝不因上下文不够而压缩内容、跳过分支、省略过渡。** 宁可少写一章完整交付，也不多写三章每章缩水。

## 对白质量自检

每写完一段对白，检查：

- [ ] 这句话是不是角色会说的？（参照角色卡的说话方式）
- [ ] 这句话是不是太书面了？（大声读一遍，嘴上说得出来吗？）
- [ ] 这句话有没有过度比喻或类比？
- [ ] 这句话有没有在替角色解释主题或传达道理？
- [ ] 这句话如果删掉，场景是否还能成立？（如果成立，考虑删掉）

## 叙述质量自检

每写完一个节点，检查：

- [ ] 有没有心理描写？（"他意识到"、"她明白了"）
- [ ] 有没有无法用视听呈现的内容？（"其实他是在XXX"）
- [ ] 互动节奏是否合格？（segments 数量是否在 1-5 之间）
- [ ] effect 是否过度使用？
- [ ] ambient 是否必要？
- [ ] progress 值是否合理？
