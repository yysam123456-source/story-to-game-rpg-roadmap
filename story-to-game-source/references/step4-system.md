# 第四步：状态系统与结局矩阵设计

## 目标

在动笔写任何 JSON 节点之前，先设计好整个游戏的变量体系和结局矩阵。先定目的地，再画路线。

## 一、主状态值 val 的设计

### 命名原则

val 的名称（variableName）应该自带戏剧张力，是这部作品气质的浓缩。玩家在界面上始终能看到这个词和它的数值变化——它不是心理学量表的标签，它是这个故事的暗语。

找到它的方法：问自己——**如果用一个词概括主角在这个故事里被撕裂的方向，这个词是什么？它应该让人一看就觉得不安、好奇、或者隐约痛。**

**好的 variableName：**

| 原作核心张力 | variableName | val 高 | val 低 |
|-------------|-------------|--------|--------|
| 一个人在真相与安全之间挣扎 | 灼见 | 看穿了一切 | 选择了不看 |
| 人对另一个人的控制与依附 | 缠度 | 深陷其中 | 挣脱了线 |
| 在道德崩塌的世界里守不守底线 | 锈蚀 | 已被侵蚀 | 还没有坏 |
| 角色在现实与幻觉之间滑动 | 失焦 | 彻底脱离 | 脚踏实地 |
| 信任一个不该信任的人 | 赌注 | 全押进去了 | 什么都没给 |
| 主角被过去追着不放 | 回声 | 被往事吞没 | 走了出来 |
| 在权力结构中的位置 | 棋温 | 已成执棋人 | 还是棋子 |
| 理性被极端情境蚕食 | 裂纹 | 理性崩碎 | 裂缝未扩 |
| 一段感情中越陷越深 | 潮位 | 已被淹没 | 潮水未至 |
| 角色在仇恨与放下之间 | 炭火 | 还在烧 | 已经凉了 |

**坏的 variableName（禁止使用）：**
勇气、信任、理性、好感、道德、情绪、状态、心理、感情、能力——这些词太通用、太平，没有作品特异性，放在任何故事里都能用等于放在哪里都不对。

### 命名自检

写出 variableName 之后，问自己三个问题：
1. 这个词是不是只有放在这部作品里才成立？（如果换到另一个故事也完全合适，说明还不够特异）
2. 这个词让人看到的时候会不会想知道"这是什么意思"？（好奇感 > 一目了然）
3. 这个词的高值和低值，是不是都不是简单的"好"和"坏"？（最好的 val 两端都有代价）

### initialVariable 的设定

初始值不应该在两端（0 或 100），应该在一个"中间偏某一侧"的位置，给玩家向两个方向发展的空间。

推荐范围：25-45 或 55-75，取决于角色在故事开始时的起点状态。

### val 变动的设计原则

1. **可感知性**：每次 val 变动必须对应玩家可以感知到的叙事差异。如果 +5 和 -5 之后剧情完全一样，这个变动就是假的。

2. **渐进性**：单次变动通常在 ±5 到 ±15 之间。超过 ±20 的变动只用于重大事件。

3. **双向性**：游戏中必须同时存在增加和减少 val 的选择。如果所有选择都只增不减（或只减不增），val 就失去了意义。

4. **阈值感**：在关键数值点（比如 30、50、70）设置条件判断，让玩家感受到"过线"的重量。

### val 的正确用途：影响体验质感，而非决定结局

**val 的主要作用不是决定玩家进入哪个结局——而是影响同一条故事线中的体验质感。**

val 应该做的事：
- **NPC 的态度变化**：val 高时某个角色更愿意帮你、说话更友善；val 低时同一个角色更冷淡、更防备
- **可见信息的差异**：val 高时角色愿意透露更多信息；val 低时某些对白被跳过或替换
- **场景细节的变化**：val 影响叙述中的环境描写倾向（同一个场景，val 高时注意到阳光，val 低时注意到阴影）
- **可用选项的差异**：某些选项只在 val 达到特定阈值时出现

val 不应该做的事：
- ❌ 作为结局的唯一判定条件（"val >= 60 → 好结局，val < 30 → 坏结局"）
- ❌ 低 val 直接等于坏结局——低 val 可能意味着角色走了一条更黑暗但同样完整的路
- ❌ 制造"刻意选低 val 就会提前死掉"的陷阱——val 低不应该导致游戏提前结束

### val 与结局的正确关系

结局应该主要由**路径（flag）**决定，val 作为辅助修饰：

```
❌ 错误设计：
routes: [
  { condition: "val >= 60", next: "ending_good" },
  { condition: "val >= 30", next: "ending_normal" },
  { condition: "default", next: "ending_bad" }
]
（完全依赖 val 数值，低 val 玩家只能得到"坏结局"）

✅ 正确设计：
routes: [
  { condition: { all: [{ flag: "saved_lin" }, { flag: "found_truth" }] }, next: "ending_true" },
  { condition: { flag: "chose_dark_path" }, next: "ending_dark" },
  { condition: { flag: "found_truth" }, next: "ending_normal" },
  { condition: "default", next: "ending_quiet" }
]
（由路径决定走向哪个结局，每个结局都是完整的故事终点）
```

val 可以在同一个结局内部产生变体——同样到达了"ending_true"，但 val 高和 val 低时 closing 的语气不同、最后一句话不同。这比用 val 把玩家分流到不同"质量"的结局要好得多。

## 二、自定义变量 variables 的设计

### 变量分类

**关系变量**：与某个角色的关系状态
```json
{ "trust_lin": 0, "respect_zhang": 1 }
```

**信息变量**：玩家是否知道某个秘密或线索
```json
{ "knows_truth": false, "found_diary": false }
```

**路线变量**：标记玩家走了哪条路径
```json
{ "route": "normal", "alliance": "none" }
```

**状态变量**：记录某种累积状态
```json
{ "wounds": 0, "reputation": "unknown" }
```

### 设计原则

1. 每个变量都必须在后续至少一个 condition 中被引用。不被使用的变量不要设计。
2. 变量名使用英文，简短且自解释：`found_key`、`trust_npc`、`route`。
3. 当前引擎中 `set` 是直接赋值，不做加减。数值增减优先使用 val。如果需要多个数值维度，用不同的变量名配合 condition 实现。

## 三、flags 的设计

flags 记录一次性事件，与 variables 的区别在于：flag 只有"有/无"两种状态。

### 适合用 flag 的场景

- 打开/检查过某个物品
- 触发过某段对话
- 到达过某个地点
- 目击过某个事件
- 做出过某个不可逆的行为

### 命名规范

使用英文，前缀+描述：
- `checked_` 检查类：`checked_drawer`、`checked_phone`
- `talked_` 对话类：`talked_to_lin`、`talked_about_past`
- `found_` 发现类：`found_letter`、`found_key`
- `did_` 行为类：`did_confess`、`did_betray`
- `seen_` 目击类：`seen_fight`、`seen_secret`

### 普通 flag vs 重要 flag（importantFlag）

启动器支持 `importantFlag`——触发时会浮出"重要决定已记录"的提示，让玩家意识到这个选择有分量。

**哪些 flag 应该标记为 important：**
- 直接决定结局走向的关键选择（如 `saved_lin`、`chose_dark_path`、`found_truth`）
- 整个游戏中只有 2-5 个，不要滥用

**哪些 flag 用普通 addFlag：**
- 日常状态记录（检查过抽屉、读过信、到过某个地点）
- 对白分支的记录（跟某人聊过某个话题）
- 大多数 flag 都应该是普通 flag

带标签的 importantFlag 可以提示决定的内容：
```json
{ "importantFlag": { "flag": "saved_lin", "label": "选择救林医生" } }
```

## 四、结局矩阵

### 设计流程

**结局不是先"设计"出来再"凑节点通向它"的——结局是分支文学大纲中每条线自然生长到的终点。**

本步骤的工作是：从第三步的分支文学大纲中提取每条线的结局方向，整理成结局矩阵，然后为整个矩阵设计状态系统（val/变量/flag/成就）。

1. **确认真结局**：对应原作正文的结尾（已在文学大纲的主线中自然到达）
2. **确认分支结局**：从每条分支线的文学大纲末尾提取——每条线发展到最后自然走向的终点就是它的结局
3. **设计普通结局**：为没有走到核心真相但合理完成故事的路径设计终点
4. **草率结局单独处理**：不在矩阵中设计，第五步写作时根据剧情自然添加
5. **设计隐藏结局**（可选）：需要特定 flag 组合或极端 val 值才能进入

### 草率结局补充规则（铁律）

草率结局是为了补充游戏性而在正式写作中添加的，与核心结局矩阵独立。它不影响真结局、普通结局、分支结局、隐藏结局的数量和分布。

**分布铁律（按篇幅缩放）：**

| 原作体量 | 前30%最少 | 30%-60%最少 | 草率结局总数参考 |
|----------|----------|------------|----------------|
| 短篇（<1万字） | 1-2个 | 1-2个 | 3-5个 |
| 中篇（1-5万字） | 2-3个 | 3-4个 | 6-10个 |
| 长篇（>5万字） | 3-5个 | 5-8个 | 10-18个 |

**故事 60% 之后不再添加草率结局**——后段的分歧通向分支结局。

长篇的草率结局应该分布在不同的剧情线上——不是主线上连着塞五六个"选错就死"，而是在不同章节、不同场景中提供不同类型的错误选择入口。

**执行方式**：在第五步分段写作中，写到故事前 30% 的位置时，必须回检：是否已经设置了至少一个草率结局入口？如果没有，停下来补。同理在写到 60% 时再检一次。

**添加时机**：每写完一个章节，回看本章是否存在"玩家做了某个明显错误的选择"的时刻。如果有，为这个选择添加一条通往草率结局的短线。

**分布建议**：
- 前期（故事前 20%）：1-2 个（"直接放弃/拒绝参与"类）
- 中前期（20%-40%）：1-2 个（"过于鲁莽/过于怯懦"类）
- 中期（40%-60%）：1-2 个（"致命错误判断"类）
- 中后期以后不再添加——走到后半段的玩家已投入大量时间，此时给草率结局是惩罚不是设计

**质量底线不变**：最少 3-5 个节点，完整因果链，必须有 description + closing。

### 结局矩阵表（不含草率结局）

```
| 结局 ID | 类型 | 标题 | 到达条件 | 内容方向 |
|---------|------|------|----------|----------|
| ending_normal | neutral | 留在房间的人 | val 30-64 | 错过了关键时刻 |
| ending_branch_dark | dark | 另一种雨 | val<30, did_betray | 走向黑暗面 |
| ending_true | true | 潮声归来 | val>=65, 关键flags | 原作结尾 |
| ending_hidden | hidden | 录音带 | 特殊flag组合 | 隐藏真相 |
```

草率结局不列入此表，在写作阶段根据剧情自然添加。

### 结局差异化要求

**绝对禁止**的情况：两个结局只是换了标题和最后一句话，中间的叙事体验完全一样。

每个结局必须在以下至少两个维度上与其他结局不同：
- 主角的最终状态（活/死/改变/未改变）
- 人物关系的最终状态（和好/决裂/陌生）
- 真相的揭露程度（知道全部/知道一半/一无所知）
- 行动的代价（付出了什么/得到了什么）
- 世界/环境的最终状态（恢复/恶化/改变）

### closing 收束语

每个结局必须写 `closing`——这是结局的最后一句余韵，玩家看完 `description` 之后读到的最后一段文字。

`description` 负责叙述结局发生了什么（事件层），`closing` 负责给玩家一个收束的感受（情绪层）。两者不能重复。字段定义见 `SCHEMA_v1.md` 中 `endings.closing`。

closing 可以是：
- **余韵**："后来你仍会在每个雨夜醒来，但这一次你知道潮声是为了把你送回岸上。"
- **反讽**："你终于做了一个正确的决定。代价只是所有人。"
- **遗憾**："多年以后你仍会记得那个没有走出去的夜晚。门一直在那里。"
- **确认**："你没有回头。你不需要回头。"
- **回声**："你再也没有收到回信。也许沉默本身就是答案。"
- **落幕**：一句非常短的、像电影最后一帧字幕的句子

不同结局的 closing 风格应该有差异——真结局的 closing 可以有温度，草率结局的 closing 应该干涩或刺痛，隐藏结局的 closing 可以留悬念。

### 结局线扩写协议（铁律）

**ending 页不是"结局本身"。ending 页只是最后的判词。真正的结局是 ending 前那一整条生活后果链。**

#### 结局线长度标准

| 结局类型 | ending 前的收束节点数 | 后果维度要求 |
|----------|-------------------|------------|
| failure | 8-10 节点 | 当场反应 + 社会后果 + 至少1个人物态度变化 |
| neutral | 10-20 节点 | 家庭 + 生计 + 关键人物 + 制度/社会 |
| dark | 10-20 节点 | 同 neutral，且必须有独特终点 |
| true | 15-30 节点 | 全维度覆盖 |
| hidden | 15-30 节点 | 全维度 + 独有信息揭示 |

#### 非原著结局推演（写作前必答）

每条非原著线在写 JSON 之前必须回答以下六个问题，答不出来不准写 ending：

1. 这个选择改变了谁的生活？
2. 三天后世界怎样了？
3. 一个月后呢？
4. 家庭/亲密关系怎样了？
5. 社会/制度/舆论怎样记录这件事？
6. 原作的核心主题在这条线上怎样被反照？

#### ending 前置检查

每个 ending 节点前的收束链必须包含：
- 至少 3 个具体生活场景（不是抽象概括）
- 至少 1 个家庭/亲密关系后果
- 至少 1 个社会/制度后果
- 至少 1 个关键物件的最终归处

#### 文学厚度自检

写完每条结局线后问一句：**"如果这条结局线单独拿出来，像不像一条完整短篇的尾声？"** 如果不像，就还没写完。

### 结局触发条件设计（铁律）

**铁律一：无论玩家怎么玩，都必须能到达某个结局。**

不允许存在"玩家走到某个分支后既不是结局也无法继续"的死路。最终的路由收束（routes）必须用 `default` 兜底，确保任何 val/flag 组合都有去处。

**铁律二：大部分结局的触发条件不能太苛刻。**

结局按触发方式分为三层：

| 层级 | 占比 | 触发方式 | 对应结局类型 |
|------|------|---------|-------------|
| 路径结局 | 60-70% | 由玩家走过的路径（flag 组合）决定 | 真结局、普通结局、分支结局 |
| 条件结局 | 15-25% | 需要特定 flag 组合 + val 辅助 | 部分分支结局 |
| 稀有结局 | 10-15% | 需要多个特定 flag 的罕见组合 | 隐藏结局、特殊结局 |

（草率结局不参与此分布，由单个错误选择在前中期直接触发）

**核心原则：结局由路径决定，不由数值决定。**

**具体设计原则：**

1. **真结局由关键 flag 组合触发。** 玩家在主线上自然获得的 2-3 个关键 flag（如 found_truth + saved_lin）指向真结局。不依赖 val 区间。

2. **分支结局由路径 flag 触发。** 玩家在关键分支点选择了不同方向，设置了不同的路径 flag（如 chose_dark_path），这个 flag 决定进入哪条结局线。

3. **val 用于结局内部的变体，不用于结局分流。** 同一个结局（如 ending_true），val 高和 val 低时可以有不同的 closing 语气、最后一句话、或 description 中的细节差异。但不用 val 把玩家送进不同"质量等级"的结局。

4. **不要制造"低 val 陷阱"。** 选择导致 val 降低的玩家不应该因此进入更短或更敷衍的结局线。低 val 意味着走了一条更黑暗/更孤独的路，但这条路应该同样充分和完整。

5. **只有隐藏结局允许苛刻条件。** 需要 3 个以上特定 flag 的罕见组合才能触发。

6. **实际测试：模拟三种玩家路径。**
   - "直觉玩家"：顺着感觉选 → 应到达真结局或普通结局
   - "极端玩家"：故意选最危险/最消极的选项 → 应到达草率结局（前中期触发）或黑暗分支结局（后期完整线路）
   - "探索玩家"：每个分支都想看看 → 应能发现隐藏结局的线索

## 五、成就设计

### 数量规则

成就数量 > 结局数量。如果有 5 个结局，至少设计 7-8 个成就。

### 成就来源

1. **选择成就**：在关键分支点做出特定选择
2. **数值成就**：val 到达关键阈值（首次超过 70、首次跌破 30）
3. **发现成就**：找到隐藏线索、进入隐藏分支
4. **路线成就**：完成特定路线组合
5. **结局成就**：到达某个结局（每个结局通常对应一个成就）
6. **行为成就**：做出反常或罕见的选择

### 命名规范

成就 ID 使用英文：`first_choice`、`low_trust`、`found_secret`、`true_ending`
成就标题和描述使用中文，风格应贴合作品气质。

## 六、RPG 扩展状态系统设计

当作品类型需要 RPG 数值面板时（修仙、无限恐怖、末世、宫斗等），需要在基础状态系统之上扩展 RPG 状态系统。

### 6.1 primaryStats 设计（可见数值）

`primaryStats` 定义在 `meta.rpg.primaryStats` 中，控制播放器顶部状态栏展示的数值。

**设计原则**：
1. **可见数值最多 5 个**：避免面板过于拥挤。
2. **每个类型有推荐变量**：参考 `GENRE_TEMPLATES.md` 中对应类型的"推荐变量"部分。
3. **区分可见与隐藏**：核心数值可见，辅助数值放入 `hiddenStats`。

**字段说明**：
```json
{
  "meta": {
    "rpg": {
      "primaryStats": [
        { "key": "qi", "label": "灵力", "type": "bar", "max": 80, "min": 0, "tone": "positive" },
        { "key": "realm", "label": "境界", "type": "text", "default": "炼气三层" }
      ]
    }
  }
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `key` | ✅ | 对应 `variables` 中的键名 |
| `label` | ✅ | UI 展示名 |
| `type` | ✅ | `"text"` / `"number"` / `"bar"` |
| `max` | ❌ | `bar` 类型最大值，默认 100 |
| `min` | ❌ | `bar` 类型最小值，默认 0 |
| `default` | ❌ | `type="text"` 时的默认值 |
| `tone` | ❌ | `"positive"` / `"danger"` / `"neutral"` |

**类型说明**：
- `text`：直接显示 `variables[key]` 的字符串值（如"炼气三层"）
- `number`：显示数值，支持加减
- `bar`：显示进度条，有最大/最小值

### 6.2 hiddenStats 设计（隐藏数值）

`hiddenStats` 定义在 `meta.rpg.hiddenStats` 中，指定可用于条件但不展示的隐藏数值键名列表。

**用途**：
- 隐藏数值可以用于条件、routes、结局判定
- 但不在普通 UI 中暴露，保持悬念和探索感

**示例**：
```json
{
  "meta": {
    "rpg": {
      "hiddenStats": ["karma", "tribulationRisk", "sectReputation"]
    }
  }
}
```

### 6.3 milestones 设计（成长里程碑）

`milestones` 是顶层里程碑定义，标记剧情中的关键成就节点。

**设计原则**：
1. **每个类型必须至少包含 1 个 `large` 里程碑**：标志玩家在剧情中的关键性进展。
2. **里程碑分为 small / medium / large 三级**：均可配置 VFX 特效。
3. **触发条件应基于变量或 flag**：而非单纯依赖 val。

**字段说明**：
```json
{
  "milestones": [
    {
      "id": "first_breakthrough",
      "name": "踏入修行",
      "condition": { "var": "realm", "op": "!=", "value": "凡人" },
      "celebration": "small",
      "vfx": "breakthrough",
      "segments": [
        { "text": "体内的灵气开始流转，你感受到了天地之间的第一缕灵韵。" }
      ]
    }
  ]
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | ✅ | 全局唯一标识 |
| `name` | ✅ | 里程碑名称 |
| `condition` | ✅ | 触发条件 |
| `celebration` | ❌ | `"small"` / `"medium"` / `"large"` |
| `vfx` | ❌ | 题材专属特效标识 |
| `segments` | ❌ | 达成时的叙事文本 |
| `changes` | ❌ | 达成时附带的状态变化 |

**celebration 级别**：
- `small`：简短通知 Toast（顶部滑入，2 秒后消失）
- `medium`：全屏半透明 overlay + 叙事文本（需点击继续）
- `large`：全屏庆祝动画 + 题材专属 VFX + 叙事文本

**检测策略**：
- 运行时在每次 `changes.apply()` 之后，遍历所有未触发的 milestones 并检查 `condition`
- `once: true` 的 milestone 触发后从检测列表中移除
- 已触发过的 milestone 通过 `saveModel.triggeredMilestones` 去重

### 6.4 delayedChanges 设计（延迟后果）

`delayedChanges` 实现"因果延迟"机制，让选择的真正影响不会立即显现。

**设计原则**：
1. **`weight=critical` 的选择必须配置至少 1 个 `delayedChanges`**：即选择的真正影响不会立即显现，而是在后续章节中以出人意料的方式爆发。
2. **延迟后果应有叙事解释**：通过 `reason` 字段增强叙事感。
3. **触发节点应合理**：不能太近（失去悬念）也不能太远（玩家忘记原因）。

**字段说明**：
```json
{
  "delayedChanges": [
    {
      "triggerNode": "node_025",
      "changes": {
        "set": { "masterTrust": -10 },
        "show": true,
        "feedback": [
          { "label": "师父信任", "delta": "-10", "tone": "negative" }
        ]
      },
      "reason": "师父发现了你隐瞒真相的事"
    }
  ]
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `triggerNode` | ✅ | 延迟到哪个节点触发 |
| `changes` | ✅ | 触发时执行的状态变化 |
| `reason` | ❌ | 解释延迟后果的原因 |

**运行时机制**：
1. 玩家做出选择后，引擎将 `delayedChanges` 记入 `saveModel.pendingDelayedChanges` 队列
2. 当玩家到达 `triggerNode` 时，引擎检查是否有匹配的 pending 变更并依次执行
3. `reason` 文本以旁白风格插入到 `triggerNode` 的 `segments` 之前

选择类型后，参考 `GENRE_TEMPLATES.md` 中对应类型的：

### 6.6 关键选择标记（importantFlag）和权重标签（weightTag）

**importantFlag**：标记影响结局走向的关键决定，播放器会短暂浮出提示"重要决定已记录"。

```json
{ "changes": { "importantFlag": { "flag": "saved_master", "label": "救下师父" } } }
```

**使用建议**：
- `weight=critical` 的选择应使用 `importantFlag` 标记
- 整个游戏中只有 2-5 个 importantFlag，不要滥用
- 带标签的 importantFlag 可以提示决定的内容

**weightTag**：选项旁边的标签文字，增强视觉提示。

```json
{
  "text": "强行突破",
  "next": "node_breakthrough",
  "weight": "critical",
  "weightTag": "关键抉择",
  "weightHint": "此选择可能导致突破成功或失败"
}
```

**使用建议**：
- 在 `choice.weightTag` 字段中显示标签（如"关键抉择"、"支线影响"）
- `weightHint` 字段在鼠标悬停/长按时显示
- 两者配合使用，增强玩家对选择重要性的感知
